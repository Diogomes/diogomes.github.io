Modelagem relacional: tabelas, chaves e relacionamentos

Diogo Gomes  ·  24 de Junho, 2026  ·  3 min de leitura

Antes de escrever qualquer consulta, todo profissional de dados precisa entender como a informação é organizada. O modelo relacional é a base de praticamente todo banco que você vai encontrar no dia a dia, e dominar seus conceitos é o primeiro passo para não cometer erros que custam caro depois.

O vocabulário básico: tabela, linha e coluna

Pense em uma planilha. Uma tabela representa uma entidade do mundo real — clientes, produtos, pedidos. Cada linha (ou registro) é uma ocorrência específica dessa entidade: um cliente em particular. Cada coluna (ou atributo) descreve uma característica: nome, e-mail, data de cadastro.

A regra de ouro é: uma tabela fala sobre uma coisa só. Misturar dados de clientes e pedidos na mesma tabela é o tipo de decisão que parece prática hoje e vira pesadelo amanhã.

Chave primária: a identidade de cada linha

A chave primária (primary key) é a coluna — ou conjunto de colunas — que identifica cada linha de forma única. Dois clientes podem ter o mesmo nome, mas nunca o mesmo id. Ela garante que você consegue apontar para um registro específico sem ambiguidade.

Na prática, costuma-se usar um número sequencial ou um identificador gerado pelo banco. O importante é que seja único e não mude com o tempo.

Chave estrangeira: conectando tabelas

A chave estrangeira (foreign key) é uma coluna que aponta para a chave primária de outra tabela. É assim que dizemos "este pedido pertence àquele cliente". Ela cria o vínculo e, além disso, garante integridade referencial: o banco impede que você cadastre um pedido para um cliente que não existe.

CREATE TABLE clientes (
  id    INTEGER PRIMARY KEY,
  nome  VARCHAR(100) NOT NULL,
  email VARCHAR(150) UNIQUE
);

CREATE TABLE pedidos (
  id         INTEGER PRIMARY KEY,
  cliente_id INTEGER NOT NULL,
  valor      DECIMAL(10,2),
  FOREIGN KEY (cliente_id) REFERENCES clientes(id)
);

Os três tipos de relacionamento

Os relacionamentos descrevem como as entidades se conectam. São três:

  • 1:1 (um para um) — cada registro de A se liga a no máximo um de B. Ex.: um usuário e seu perfil estendido. É raro e muitas vezes pode virar uma única tabela.
  • 1:N (um para muitos) — o mais comum. Um cliente tem vários pedidos, mas cada pedido pertence a um único cliente. A chave estrangeira fica no lado "muitos".
  • N:N (muitos para muitos) — alunos cursam várias disciplinas e cada disciplina tem vários alunos. Aqui surge a tabela de junção.

O caso N:N e a tabela de junção

Um banco relacional não consegue representar N:N diretamente. A solução é criar uma terceira tabela, chamada de tabela de junção (ou associativa), que guarda os pares. Ela tem duas chaves estrangeiras, uma para cada lado.

CREATE TABLE matriculas (
  aluno_id     INTEGER REFERENCES alunos(id),
  disciplina_id INTEGER REFERENCES disciplinas(id),
  PRIMARY KEY (aluno_id, disciplina_id)
);

Visualizando o relacionamento

Um diagrama simples ajuda a fixar a relação 1:N entre clientes e pedidos:

  clientes                pedidos
 +--------+              +-------------+
 | id (PK)|◄────────────┤ cliente_id  |
 | nome   |   1     N    | id (PK)     |
 | email  |              | valor       |
 +--------+              +-------------+
Modelar bem não é decorar tipos de chave: é perguntar "do que cada tabela fala?" e "como elas se conectam?". Quem responde isso com clareza projeta esquemas que aguentam o crescimento.

É comum em entrevista pedir para modelar um cenário simples — uma loja, uma biblioteca — e identificar os relacionamentos. No dia a dia isso aparece toda vez que você precisa decidir onde uma nova coluna deve morar ou por que um JOIN está duplicando linhas.

Conclusão

O modelo relacional gira em torno de poucos conceitos: tabelas como entidades, chaves que identificam e conectam registros, e relacionamentos que estruturam o todo. Com essa base sólida, escrever consultas e evoluir o esquema deixa de ser tentativa e erro. O próximo passo natural é aprender a consultar esses dados de forma eficiente.
