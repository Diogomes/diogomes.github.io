Push notifications: como funcionam de ponta a ponta

Diogo Gomes  ·  8 de Junho, 2026  ·  3 min de leitura

Push notifications parecem mágica: o servidor "manda" uma mensagem e ela aparece no celular mesmo com o app fechado. Por baixo há um fluxo bem definido envolvendo permissão, um token de dispositivo e os serviços da Apple (APNs) e do Google (FCM). Entender esse caminho de ponta a ponta — e por que a entrega nunca é 100% garantida — é esperado de um sênior.

O fluxo de ponta a ponta

1. App pede permissao ao usuario
2. SO/serviço devolve um TOKEN unico do dispositivo
3. App envia o token ao SEU backend, que o guarda
4. Backend manda a mensagem + token para APNs (Apple) ou FCM (Google)
5. APNs/FCM entrega ao dispositivo certo
6. SO exibe a notificacao (ou acorda o app)

A analogia: o token é como o endereço de entrega do dispositivo. Seu servidor não entrega o pacote em mãos — ele dá o pacote e o endereço para os Correios (APNs/FCM), e estes fazem a entrega física.

Permissão: o usuário decide

No iOS (e a partir do Android 13) o app precisa pedir permissão para notificar. Se o usuário recusa, não há push — ponto. Por isso pede-se a permissão no momento certo, explicando o valor, e não logo na primeira tela. É comum em entrevista pedir para descrever esse passo e suas implicações de UX e de taxa de entrega.

O token do dispositivo

Após a permissão, o serviço do SO gera um token que identifica aquele app naquele aparelho. Pontos cruciais:

  • O token pode mudar (reinstalação, troca de aparelho, atualização) — seu backend precisa atualizá-lo.
  • Um usuário pode ter vários tokens (vários dispositivos).
  • Token inválido/expirado deve ser removido do banco, ou você desperdiça envios.

APNs e FCM: os carteiros

Seu backend nunca fala direto com o dispositivo. Ele autentica-se no APNs (Apple Push Notification service) para iOS ou no FCM (Firebase Cloud Messaging) para Android, envia a mensagem com o token de destino, e o serviço cuida da entrega — inclusive de acordar o app se necessário.

  [ Seu backend ]
        |  (mensagem + token + auth)
        v
   [ APNs / FCM ]   <- fila best-effort
        |
        v
   [ Dispositivo ] --> SO exibe notificacao

Notificação vs data message

Há dois tipos de payload, e a diferença cai em entrevista:

TipoO que fazApp fechado?
NotificationSO exibe o alerta automaticamente (título/corpo)Aparece sem o app rodar
Data messageEntrega dados ao app para ele decidir o que fazerDepende do SO acordar o app; pode ser adiado

Um payload pode combinar os dois. A escolha define se o SO mostra sozinho ou se seu código processa (ex.: atualizar um cache em silêncio).

Por que a entrega não é garantida

Push é best-effort, não garantido. A mensagem pode não chegar (ou atrasar) porque:

  • O dispositivo está offline; APNs/FCM guardam por um tempo e podem descartar.
  • O SO está economizando bateria (Doze/App Standby no Android) e adia o trabalho.
  • O sistema agrupa ou limita data messages de apps "inativos".
  • O usuário desativou notificações.

Por isso push nunca deve ser o único canal de uma informação crítica: ele avisa, mas a fonte da verdade está no servidor, e o app deve revalidar ao abrir.

Trate push como um lembrete, não como um canal confiável de dados. Se a informação for indispensável, o app precisa conseguir obtê-la sozinho ao abrir — o push só apressa a descoberta.

Conclusão

O fluxo permissão → token → backend → APNs/FCM → dispositivo, a diferença entre notification e data message, e a natureza best-effort da entrega são o núcleo do que se espera saber sobre push. Quem entende que o serviço da plataforma é o carteiro — e que o app não pode depender 100% dele — projeta notificações que funcionam sem criar bugs de dados.
