Estado e dados no mobile: do local ao gerenciamento global

Diogo Gomes  ·  17 de Junho, 2026  ·  3 min de leitura

Conforme um app cresce, a pergunta "onde guardo este dado?" aparece o tempo todo. A resposta depende de quanto tempo o dado precisa viver, quem mais o usa e qual o seu formato. Saber separar estado efêmero de tela do estado do app, e escolher entre key-value, banco local e store global, é o que distingue um pleno na hora de modelar dados.

Estado efêmero de tela vs estado do app

Nem todo dado tem a mesma importância. O estado efêmero pertence a uma tela e morre com ela: se um campo está expandido, qual aba está selecionada, o texto sendo digitado. Já o estado do app é compartilhado e duradouro: o usuário logado, o tema escolhido, o carrinho de compras.

A analogia: estado de tela é o rascunho num post-it que você joga fora; estado do app é o caderno que vários setores consultam. Tratar tudo como estado global infla o app desnecessariamente; tratar tudo como local força você a "passar o dado de mão em mão" entre telas.

Armazenamento local: key-value

Para pedaços pequenos e simples de configuração, todas as plataformas oferecem um armazenamento chave-valor: UserDefaults (iOS), SharedPreferences (Android), AsyncStorage (React Native). É como uma gavetinha de bilhetes: ótimo para "tema = escuro" ou "já viu o onboarding = sim", péssimo para listas grandes ou dados relacionados.

// key-value: pequeno e simples
prefs.set("tema", "escuro")
prefs.set("onboardingVisto", true)

let tema = prefs.get("tema")

Armazenamento local: banco

Quando os dados são muitos, estruturados e relacionados (uma lista de pedidos, mensagens, produtos em cache), você precisa de um banco local: SQLite na base, com camadas como Room (Android), Core Data (iOS) ou bibliotecas multiplataforma. Bancos permitem consultas, filtros e relações — coisa que a gaveta de bilhetes não faz.

Memória (estado de tela)Dados temporários da UIAlgo que precisa sobreviver ao fechar

Cache: nem fonte, nem lixo

Cache é uma cópia local de dados que vivem no servidor, mantida para ganhar velocidade e funcionar offline. A regra de ouro: cache pode ser descartado a qualquer momento sem perda real, pois a fonte da verdade está no backend. Confundir cache com armazenamento permanente leva a dados desatualizados ou perda do que deveria ser persistido. O cache é a base do offline-first.

Quando partir para estado global / store

Se você se pega passando o mesmo dado por cinco telas só para chegar na sexta ("prop drilling"), ou se vários pontos do app precisam reagir à mesma mudança (logout, troca de tema), é hora de um estado global: um store (Redux, Zustand, ViewModels compartilhados, Provider/Riverpod, ObservableObject). Ele centraliza o estado do app e notifica quem se interessa.

// store global: uma fonte, varios ouvintes
store.usuario = login(...)   // muda em um lugar
// telas inscritas re-renderizam sozinhas

É comum em entrevista pedir para você decidir "onde guardar X" em um cenário dado — e a resposta forte justifica a escolha pelo tempo de vida e pelo alcance do dado, não por hábito.

Não pergunte "qual ferramenta de estado usar", pergunte "quanto tempo este dado vive e quem precisa dele". A ferramenta cai naturalmente a partir daí.

Conclusão

Estado efêmero na memória da tela, configurações pequenas em key-value, dados estruturados em banco local, cache como cópia descartável e estado compartilhado num store global: cada dado tem seu lugar. Modelar isso com clareza deixa o app rápido, previsível e pronto para crescer — inclusive para funcionar sem rede.
