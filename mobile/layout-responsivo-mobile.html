Layout que se adapta: densidade, safe areas e telas diferentes

Diogo Gomes  ·  20 de Junho, 2026  ·  3 min de leitura

Existem celulares minúsculos, telas enormes, notches, barras de status e tablets — e seu layout precisa ficar bom em todos. O segredo não é desenhar em pixels, mas pensar em unidades lógicas, respeitar as áreas seguras e deixar o layout se adaptar sozinho. É um dos primeiros temas em que um júnior tropeça e um dos primeiros que entrevistadores cobram.

Pixel físico vs pixel lógico

Dois celulares podem ter o mesmo tamanho físico mas números de pixels muito diferentes: um tem tela "normal", outro "retina/alta densidade" com o dobro de pixels no mesmo espaço. Se você posicionasse tudo em pixels físicos, o botão de 100px ficaria gigante numa tela e minúsculo na outra.

Por isso as plataformas usam unidades lógicas: dp (density-independent pixels, Android) e pt (points, iOS). Você descreve "quero 16 unidades de espaço" e o sistema multiplica pela densidade da tela para chegar nos pixels físicos reais.

DensidadeFator16 dp/pt viram
mdpi / @1x1x16 px físicos
xhdpi / @2x2x32 px físicos
xxhdpi / @3x3x48 px físicos

A analogia: dp/pt são como "centímetros" do mundo mobile — uma medida que significa o mesmo tamanho aparente em qualquer tela. Pixels físicos são como contar grãos de areia, que variam de praia para praia.

Por que medir em dp/pt

Medindo em dp, um espaçamento de 8 dp parece igual no celular do estagiário e no topo de linha. O sistema cuida da conversão. É comum em entrevista pedir para explicar a diferença entre px e dp e por que nunca se deve cravar pixels físicos — a resposta esperada é exatamente essa independência de densidade.

A armadilha do hardcode de pixels

// RUIM: assume largura fixa em pixels
imagem.largura = 320      // estoura em tela pequena

// BOM: relativo ao espaço disponível
imagem.largura = "100%"   // ou usa constraints/flex
imagem.altura  = aspecto(16, 9)

Cravar 320 de largura quebra no aparelho de 4 polegadas e desperdiça espaço no tablet. No dia a dia esse bug aparece como texto cortado, botão fora da tela ou layout que "estoura" em certos modelos.

Safe areas: notch, status bar e home indicator

A tela retangular nem toda é utilizável. No topo há a barra de status (relógio, bateria) e o notch/ilha; embaixo, a barra de gestos ou home indicator. A safe area é a região onde seu conteúdo não fica escondido atrás desses elementos.

+---------------------+
| [status bar / notch]|  <- nao coloque botoes aqui
|                     |
|   AREA SEGURA       |
|   (seu conteudo)    |
|                     |
| [home indicator]    |  <- nem aqui
+---------------------+

Ignorar a safe area faz o título sumir atrás do notch ou o botão de ação ficar embaixo da barra de gestos, onde o usuário não consegue tocar direito.

Do celular pequeno ao tablet

Adaptar não é só esticar. Use constraints (iOS Auto Layout), flexbox (React Native) ou widgets flexíveis (Flutter, ConstraintLayout no Android) para que os elementos se reorganizem conforme o espaço. Em tablets, é comum mudar o layout em si: uma lista que vira lista + detalhe lado a lado (master-detail).

  • Proporções, não tamanhos fixos: "ocupe metade da largura" em vez de "240 dp".
  • Breakpoints: trocar de layout de coluna única para duas colunas acima de certa largura.
  • Texto que cresce: respeitar a fonte aumentada pelo usuário (acessibilidade).
Você não desenha para um aparelho; desenha regras de adaptação. Pense em "como isto se comporta quando o espaço encolhe ou dobra", não em "quantos pixels tem aqui".

Conclusão

Medir em dp/pt, respeitar as safe areas e usar constraints/flex em vez de pixels cravados é o tripé do layout responsivo mobile. Quem internaliza a diferença entre pixel físico e lógico evita os bugs visuais mais frequentes e entrega telas que funcionam do menor celular ao maior tablet.
