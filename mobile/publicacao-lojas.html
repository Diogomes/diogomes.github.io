Publicação nas lojas: build, assinatura, review e versionamento

Diogo Gomes  ·  6 de Junho, 2026  ·  3 min de leitura

Escrever o app é metade do trabalho; colocá-lo na mão dos usuários é a outra. Publicar nas lojas envolve gerar um build de release assinado, passar pelo review da Apple ou do Google, versionar corretamente e atualizar sem quebrar quem já tem o app instalado. Dominar esse pipeline de release é responsabilidade típica de um sênior.

Build de debug vs release

O build que você roda no dia a dia (debug) é diferente do que vai para a loja (release). O debug prioriza velocidade de desenvolvimento e diagnóstico; o release prioriza tamanho, performance e segurança.

AspectoDebugRelease
Otimização de códigoMínima (compila rápido)Máxima (mais rápido, menor)
Ofuscação/minificaçãoDesligadaLigada (ex.: R8/ProGuard)
Logs e debugHabilitadosRemovidos/desligados
AssinaturaChave de debug automáticaCertificado/keystore de produção
DestinoSeu aparelho/emuladorApp Store / Play Store

Assinatura de código: o que e por quê

Todo app instalado é assinado criptograficamente. A assinatura prova que o pacote veio de você e não foi adulterado no caminho. No Android isso é uma keystore (sua chave privada); no iOS, certificados e provisioning profiles emitidos pela Apple.

A analogia: a assinatura é o lacre de segurança de um remédio. O SO só instala se o lacre bater, e só aceita uma atualização se for assinada pela mesma chave da versão anterior — caso contrário, recusa.

Perder a chave de assinatura de produção é catastrófico: sem ela você não consegue publicar atualizações do mesmo app, e usuários ficam presos na versão antiga. Guarde-a como guarda um cofre — backup seguro e acesso restrito.

O processo de review

Você não publica direto: envia o build e ele entra em review. A Apple faz revisão majoritariamente manual e mais rigorosa (diretrizes de conteúdo, privacidade, uso de APIs); a Google combina checagens automáticas com revisão, geralmente mais rápida. Motivos comuns de rejeição: pedir permissões sem justificativa, crashes, política de privacidade ausente, ou tentar burlar pagamentos da plataforma. Planeje prazo de review no cronograma de lançamento.

Versionamento: nome vs número de build

Há dois números e eles servem a públicos diferentes:

  • Version name / marketing version (ex.: 2.3.0): o que o usuário vê. Semântico, legível.
  • Build number / version code (ex.: 147): um inteiro que sempre cresce, usado pela loja para saber qual é mais novo.
versionName "2.3.0"   // o usuario ve
versionCode 147       // a loja compara; nunca repita nem diminua

É comum em entrevista pedir a diferença entre os dois — a chave é: o build number é a identidade técnica monotônica; o version name é a comunicação humana.

Atualizar sem quebrar quem já tem o app

Usuários atualizam em ritmos diferentes; por dias ou semanas, versões antigas e novas coexistem. Cuidados essenciais:

  • Compatibilidade de API: o backend precisa atender versões antigas e novas ao mesmo tempo; evite quebrar contratos.
  • Migração de dados local: se o esquema do banco mudou, migre os dados de quem atualizou sem perder nada.
  • Forçar atualização com parcimônia: só bloqueie versões realmente incompatíveis.

Rollout gradual

Em vez de liberar para 100% de uma vez, as lojas permitem rollout em fases (ex.: 5% → 20% → 50% → 100%). Assim, se a nova versão tiver um crash grave, você o vê nas métricas e pausa antes de atingir toda a base. É a rede de segurança do lançamento — combine com monitoramento de crashes para reagir a tempo.

Conclusão

Publicar é um pipeline: gerar release otimizado, assinar com a chave de produção (e protegê-la), passar pelo review, versionar com nome legível e build number crescente, e atualizar respeitando quem está em versões antigas — tudo idealmente com rollout gradual. Quem domina esse caminho entrega valor ao usuário com segurança, sem transformar um lançamento em incêndio.
