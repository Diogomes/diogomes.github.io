Performance mobile: jank, listas e consumo de bateria

Diogo Gomes  ·  13 de Junho, 2026  ·  3 min de leitura

Um app pode estar "funcionando" e ainda assim parecer ruim: rola travado, demora a responder, esquenta o celular e drena a bateria. Performance mobile é entender que você tem um orçamento de tempo por frame, recursos finitos de bateria e rede, e que listas e imagens são os vilões mais comuns. Esse é o tipo de problema que um pleno precisa diagnosticar e corrigir.

O orçamento de 16ms por frame

Para uma animação fluida a 60fps, a tela é redesenhada 60 vezes por segundo — ou seja, você tem cerca de 16 milissegundos para preparar cada frame. Em telas de 120Hz, são ~8ms. Se um frame estourar esse orçamento, ele é "perdido": a imagem não atualiza a tempo e o usuário vê um solavanco.

A analogia: é uma esteira de produção que precisa entregar uma peça a cada 16ms. Se uma estação demora demais, a linha inteira engasga.

O que é "jank"

Jank é exatamente esse engasgo visual: frames perdidos durante rolagem ou animação, percebidos como travadinhas. Quase sempre a causa é trabalho pesado rodando na thread errada na hora errada.

Fora da main/UI thread

A main thread (ou UI thread) é a única que desenha a tela e processa toques. Se você fizer nela algo demorado — ler um arquivo grande, processar JSON, decodificar imagem, consultar banco — ela não consegue desenhar o próximo frame e o app congela. A regra: trabalho pesado vai para threads de fundo (coroutines, GCD/async, isolates, workers); só o resultado volta para a UI.

// RUIM: trava a UI
let dados = baixarEParsearTudo()   // na main thread
mostrar(dados)

// BOM
async {
  let dados = baixarEParsearTudo() // em background
  naMainThread { mostrar(dados) }  // so o resultado
}

É comum em entrevista pedir para identificar por que uma tela "trava ao rolar" — a resposta forte aponta trabalho síncrono na main thread.

Listas longas: reciclagem e virtualização

Uma lista de 10 mil itens não pode criar 10 mil views. A solução é reciclar: manter apenas as views visíveis (mais uma margem) e reaproveitá-las conforme rola, trocando só o conteúdo. RecyclerView (Android), UICollectionView/diffable (iOS), FlatList (React Native) e ListView.builder (Flutter) fazem isso. Renderizar tudo de uma vez é a causa número um de jank em listas.

Imagens: tamanho e cache

  • Redimensione antes de exibir: carregar uma foto de 4000px num thumbnail de 100px desperdiça memória e tempo de decodificação.
  • Cache em memória e disco: não rebaixe e redecodifique a mesma imagem a cada rolagem (bibliotecas como Glide, Coil, SDWebImage, Kingfisher cuidam disso).
  • Decodifique fora da main thread: decodificação de imagem é cara.

Bateria e rede são finitas

Cada acesso à rede e cada wake-up do rádio consome bateria. Boas práticas:

  • Agrupar requisições em vez de muitas pequenas e frequentes.
  • Respeitar o modo de economia de bateria e fazer trabalho pesado em segundo plano via APIs do SO (WorkManager, BackgroundTasks).
  • Não fazer polling agressivo; preferir push quando possível.
  • Liberar sensores (GPS, câmera) assim que não forem necessários.

Causas comuns e correções

SintomaCausa provávelCorreção
Lista trava ao rolarTrabalho na main thread / sem reciclagemBackground thread + lista virtualizada
App esquenta e gasta bateriaPolling/loop/GPS sempre ligadoAgrupar trabalho, liberar sensores
Imagens piscam/lentasSem cache, tamanho erradoCache + redimensionar
Toque demora a responderMain thread bloqueadaMover I/O e parse para fundo
A main thread é sagrada: ela só deve desenhar e responder a toques. Todo o resto — rede, banco, parse, imagem — pertence ao fundo. Internalize isso e a maioria do jank desaparece.

Conclusão

Performance mobile se resume a respeitar o orçamento de 16ms, manter a main thread livre, reciclar listas longas, tratar imagens com tamanho e cache, e gastar bateria e rede como recursos preciosos. Diagnosticar onde o tempo está sendo gasto, e mover o peso para o lugar certo, é o que transforma um app funcional em um app que parece bom.
