Arquitetura mobile: MVVM, camadas e testabilidade

Diogo Gomes  ·  10 de Junho, 2026  ·  3 min de leitura

Quando um app cresce, a lógica de negócio espalhada dentro das telas vira um nó impossível de testar e manter. Arquitetura mobile é a disciplina de separar responsabilidades — UI de um lado, regras de negócio de outro, acesso a dados em um terceiro — para que cada parte seja simples, substituível e testável. É exatamente o tipo de decisão que se espera de um sênior.

Por que separar UI da lógica

A tela (View) deveria ser "burra": mostrar dados e encaminhar eventos. Se ela também decide regras de negócio, faz chamadas de rede e formata dados, fica impossível testá-la sem subir a UI inteira, e qualquer mudança vira risco. Separar permite testar a lógica sem tela, trocar a UI sem reescrever regras, e dividir o trabalho no time.

A analogia: um restaurante. O garçom (View) anota e serve, mas não cozinha. A cozinha (lógica) prepara o prato sem se importar com qual mesa. O estoque (dados) abastece a cozinha. Cada um troca de pessoa sem parar o restaurante.

MVC, MVVM e MVI em uma pincelada

São padrões que organizam quem fala com quem entre tela, lógica e dados.

AspectoMVCMVVM
PeçasModel, View, ControllerModel, View, ViewModel
Quem prepara dados p/ telaController (some confunde com a View)ViewModel, exposto como estado observável
Acoplamento da ViewTende a inchar (Massive View Controller)View observa o ViewModel; baixo acoplamento
Testabilidade da lógicaDifícil quando mistura com a ViewAlta: ViewModel testável sem UI

MVI (Model-View-Intent) leva o MVVM adiante com um estado único e imutável e um fluxo unidirecional: a View emite intenções, elas viram um novo estado, a View se redesenha. Frameworks declarativos (Compose, SwiftUI, React) combinam naturalmente com essa ideia.

Camadas: apresentação, domínio, dados

+------------------------------------+
|  APRESENTACAO  (View + ViewModel)  |  o que o usuario ve
+------------------------------------+
|  DOMINIO       (casos de uso,      |  regras de negocio
|                 entidades)         |  puras, sem framework
+------------------------------------+
|  DADOS         (repositorios,      |  de onde vem o dado
|                 API, banco, cache) |  (rede/local)
+------------------------------------+
   dependencia aponta para baixo -->

A regra de ouro: dependências apontam para dentro/baixo. O domínio não conhece a UI nem a API concreta — ele é o coração estável do app.

O repositório

O repositório é a fachada entre domínio e dados: ele decide se um dado vem da rede, do banco local ou do cache, e entrega isso pronto para o domínio. A View nunca sabe (nem deve saber) a origem. Isso é o que viabiliza estratégias como offline-first sem espalhar essa lógica pela tela.

interface UsuarioRepository {
  fun obter(id): Usuario
}
// implementacao decide: cache? local? rede?
// dominio so chama obter(id) e confia

Injeção de dependência

Em vez de uma classe criar suas próprias dependências (val api = ApiReal()), ela recebe de fora. Assim, em produção você injeta a implementação real; em teste, uma falsa (fake/mock). Ferramentas como Hilt/Dagger, Koin, Swinject ou injeção manual via construtor fazem isso. É comum em entrevista pedir para explicar por que DI ajuda nos testes — a resposta é exatamente poder substituir o real pelo falso.

Por que tudo isso torna o app testável

  • ViewModel/casos de uso: testáveis com testes unitários rápidos, sem subir tela.
  • Repositório atrás de interface: você injeta um fake e testa sem rede real.
  • Domínio puro: sem dependência de framework, roda em qualquer ambiente de teste.
Arquitetura não é sobre diagramas bonitos; é sobre conseguir trocar uma peça (a UI, o banco, a API) sem reescrever as outras — e provar que cada uma funciona isoladamente com um teste.

Conclusão

Separar apresentação, domínio e dados, esconder a origem dos dados atrás de um repositório, usar MVVM/MVI para libertar a View de regras e injetar dependências para tornar tudo substituível: esse é o kit de arquitetura que mantém um app grande sustentável e testável. O sênior não escolhe um padrão por moda, mas pela testabilidade e clareza de responsabilidades que ele traz.
