TDD na prática: red, green, refactor

Diogo Gomes  ·  28 de Maio, 2026  ·  1 min de leitura

TDD — Test-Driven Development — inverte a ordem intuitiva: você escreve o teste antes do código. Parece contraintuitivo, mas é uma das técnicas que mais melhora o design e a confiança no que se entrega.

O ciclo: Red, Green, Refactor

  1. Red: escreva um teste que falha — ele descreve o comportamento que ainda não existe.
  2. Green: escreva o mínimo de código para o teste passar. Sem firula.
  3. Refactor: melhore o código com a rede de segurança do teste verde.
// 1. Red — o teste falha porque a função não existe
test('soma dois números', () => {
  expect(soma(2, 3)).toBe(5);
});

// 2. Green — o mínimo para passar
function soma(a, b) { return a + b; }

// 3. Refactor — agora com segurança para evoluir

Por que funciona

  • Design guiado pelo uso: você projeta a API pensando em quem a chama.
  • Cobertura natural: o código nasce testado, sem "depois eu escrevo o teste".
  • Feedback curto: erros aparecem em segundos, não em produção.
  • Documentação viva: os testes descrevem o comportamento esperado.
TDD não é sobre testes — é sobre design. Os testes são o efeito colateral feliz.

Quando pesar a mão

TDD brilha em lógica de negócio e regras complexas. Para protótipos descartáveis ou código de exploração, pode ser exagero. O segredo é usar a ferramenta certa para o contexto — não transformar TDD em dogma.

Conclusão

Praticar o ciclo red-green-refactor treina o time a pensar em comportamento e a manter o código sempre sob uma rede de segurança. É um investimento que se paga na primeira refatoração tranquila.
