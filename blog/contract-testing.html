Contract testing: garantindo o contrato entre serviços

Diogo Gomes  ·  20 de Maio, 2026  ·  1 min de leitura

Em arquiteturas com muitos serviços, o maior risco não está dentro de cada serviço — está no contrato entre eles. Um time muda o formato de uma resposta e, sem perceber, quebra três consumidores. Contract testing existe para evitar exatamente isso.

O problema dos testes E2E entre serviços

Subir todos os serviços juntos para testar a integração é lento, caro e frágil. Contract testing troca isso por uma verificação leve: cada lado prova que respeita o contrato acordado, sem precisar do outro no ar.

Consumer-driven contracts

Com ferramentas como o Pact, o consumidor declara o que espera da API. Isso gera um contrato que o provedor verifica no próprio pipeline.

// Lado consumidor: "espero este formato"
await provider.addInteraction({
  state: 'usuário 1 existe',
  uponReceiving: 'um pedido pelo usuário 1',
  withRequest:  { method: 'GET', path: '/users/1' },
  willRespondWith: {
    status: 200,
    body: { id: 1, name: like('Diogo') }
  }
});

O fluxo

  1. O consumidor define expectativas e gera o pact (o contrato).
  2. O contrato é publicado num broker.
  3. O provedor roda o contrato contra sua API real e prova que o respeita.
  4. Quebrou o contrato? O pipeline do provedor falha — antes de chegar em produção.
Contract testing dá o melhor dos dois mundos: a confiança da integração com a velocidade do teste de unidade.

Conclusão

Para sistemas distribuídos, contratos versionados e verificados automaticamente são uma rede de segurança poderosa. Eles transformam "será que quebrei alguém?" em uma resposta objetiva dada pelo pipeline.
