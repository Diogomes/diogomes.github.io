Page Object Model: uma suíte E2E que escala

Diogo Gomes  ·  20 de Junho, 2026  ·  5 min de leitura

Toda suíte E2E começa pequena e organizada. Com o tempo, sem disciplina, ela vira uma teia de seletores duplicados em que mudar um botão quebra trinta testes. O Page Object Model (POM) é o padrão que evita esse caos.

A ideia central

Cada página (ou componente) da aplicação vira uma classe que encapsula como interagir com ela. O teste descreve o que fazer; o Page Object sabe onde estão os elementos.

class LoginPage {
  constructor(page) { this.page = page; }

  async open()        { await this.page.goto('/login'); }
  async login(u, p) {
    await this.page.getByLabel('E-mail').fill(u);
    await this.page.getByLabel('Senha').fill(p);
    await this.page.getByRole('button', { name: 'Entrar' }).click();
  }
}

O teste fica limpo e fala a linguagem do negócio:

const login = new LoginPage(page);
await login.open();
await login.login('qa@exemplo.com', 's3nh4');
await expect(page.getByText('Bem-vindo')).toBeVisible();

Regras que mantêm o POM saudável

  • Sem asserts dentro do Page Object — ele age sobre a página; quem valida é o teste.
  • Um seletor, um lugar — mudou o HTML? Ajusta em uma classe só.
  • Métodos por intenção (login()), não por mecânica (clicarBotao()).
  • Componentes reutilizáveis (menu, modal) viram objetos próprios, compostos nas páginas.
Quando um redesign quebra a suíte inteira em poucos arquivos — e não em todos — o POM está cumprindo seu papel.

Conclusão

POM não é burocracia: é o que faz a automação escalar junto com o produto. Combinado a localizadores acessíveis e fixtures, mantém a suíte legível e barata de manter mesmo depois de centenas de testes.
