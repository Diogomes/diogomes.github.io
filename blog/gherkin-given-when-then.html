Gherkin e Given-When-Then: especificação que vira teste

Diogo Gomes  ·  24 de Junho, 2026  ·  7 min de leitura

Gherkin é a linguagem por trás do BDD: um conjunto pequeno de palavras-chave que deixa um cenário de teste legível por qualquer pessoa — do Product Owner ao desenvolvedor — e, ao mesmo tempo, executável por ferramentas como Cucumber, Behave (Python), SpecFlow (.NET) ou Behat (PHP). É o ponto onde a especificação e o teste viram o mesmo documento.

As palavras-chave

Toda a gramática cabe em poucas palavras. O resto é texto livre que você escolhe:

  • Funcionalidade (Feature) — o agrupador; descreve a capacidade do sistema.
  • Cenário (Scenario) — um exemplo concreto de comportamento.
  • Dado (Given) — o contexto / pré-condição. O mundo antes da ação.
  • Quando (When) — a ação que dispara o comportamento. Uma só por cenário, idealmente.
  • Então (Then) — o resultado observável que se verifica.
  • E / Mas (And / But) — encadeiam passos do mesmo tipo, mantendo a leitura fluida.

A estrutura Given-When-Then

O coração do Gherkin é separar contexto → ação → resultado. Essa tríade força o autor a pensar em comportamento, não em telas:

# language: pt
Funcionalidade: Autenticação de usuário

  Cenário: Login com credenciais válidas
    Dado que estou na página de login
    E possuo uma conta com e-mail "ana@exemplo.com"
    Quando informo as credenciais corretas
    Então devo ver o painel inicial
    E devo ver a saudação "Olá, Ana"
Regra de ouro: Given não tem ação, When é uma ação só, Then não tem efeito colateral — só verifica. Se um cenário tem três "Quando", provavelmente são três cenários disfarçados.

Background: o contexto que se repete

Quando vários cenários compartilham os mesmos Given, o Background (Contexto) evita repetição — ele roda antes de cada cenário do arquivo:

  Contexto:
    Dado que estou logado como administrador
    E existe um produto "Caderno" no catálogo

Scenario Outline: um cenário, muitos dados

Para testar a mesma regra com várias entradas — o caso clássico de partição de equivalência e valor-limite — usa-se o Esquema do Cenário com uma tabela de Exemplos:

  Esquema do Cenário: Desconto por faixa de valor
    Dado um carrinho com <valor> reais
    Quando o cliente finaliza a compra
    Então o total cobrado deve ser <total> reais

    Exemplos:
      | valor | total |
      | 100   | 100   |
      | 101   | 90.9  |
      | 150   | 135   |

Cada linha da tabela vira um cenário independente. É a forma mais limpa de cobrir limites (o 100 que não ganha desconto e o 101 que ganha) sem copiar e colar.

Do texto ao código: step definitions

O Gherkin sozinho não testa nada — ele é "colado" ao sistema pelos step definitions: funções que casam com cada passo por expressão e executam a automação de verdade. Em Cucumber com JavaScript:

const { Given, When, Then } = require('@cucumber/cucumber');
const { expect } = require('@playwright/test');

Given('um carrinho com {int} reais', async function (valor) {
  this.carrinho = criarCarrinho(valor);
});

When('o cliente finaliza a compra', async function () {
  this.total = this.carrinho.finalizar();
});

Then('o total cobrado deve ser {float} reais', async function (esperado) {
  expect(this.total).toBeCloseTo(esperado, 2);
});

Repare que o mesmo step um carrinho com {int} reais serve a todas as linhas do Exemplos. Steps bem escritos são reutilizáveis — esse é o ganho de manutenção do BDD.

Declarativo > imperativo

O erro mais comum é escrever cenários imperativos, descrevendo cada clique. Eles quebram a cada ajuste de tela e ninguém do negócio consegue ler:

Imperativo (frágil)Declarativo (robusto)
Quando preencho o campo "email" com "ana@x.com"
E preencho o campo "senha" com "123"
E clico no botão "#btn-login"		 Quando faço login com credenciais válidas

O "como preencher o formulário" pertence ao step definition (e ao Page Object), não ao Gherkin. O cenário descreve a intenção.

Anti-patterns para vigiar

  • Cenário-novela: dez passos cobrindo um fluxo inteiro. Um cenário = um comportamento.
  • Detalhe de UI no Then: verificar cor de botão ou CSS. Verifique resultado de negócio.
  • Acoplamento entre cenários: o cenário 2 depender do estado deixado pelo cenário 1. Cada um deve ser independente.
  • Gherkin sem conversa: escrever os .feature sozinho, depois da implementação. Aí é só relatório enfeitado — perde-se o valor de descoberta do BDD.

Conclusão

Gherkin é simples de propósito: a restrição de vocabulário é o que mantém o cenário legível para todos e estável no tempo. Dominar Given-When-Then, Background e Scenario Outline — e saber escrever de forma declarativa — é o que transforma uma suíte de BDD em documentação viva em vez de mais um conjunto de testes frágeis.
