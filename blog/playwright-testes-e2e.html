Playwright na prática: testes E2E modernos

Diogo Gomes  ·  18 de Junho, 2026  ·  2 min de leitura

Depois de anos automatizando com Selenium, migrar parte das suítes para o Playwright mudou a forma como o time encara testes end-to-end. Não é só uma ferramenta nova — é uma forma mais estável e rápida de validar o comportamento real da aplicação.

Por que Playwright?

O grande diferencial é o auto-waiting: o Playwright espera automaticamente o elemento ficar visível, habilitado e estável antes de interagir. Isso elimina boa parte dos sleep() e dos testes "flaky" que tanto atrapalham a confiança na suíte.

  • Multi-browser real: Chromium, Firefox e WebKit com a mesma API.
  • Execução paralela por padrão, com isolamento por contexto de browser.
  • Trace Viewer: grava cada passo com screenshots, DOM e network — depurar uma falha de CI vira algo trivial.
  • Auto-wait e web-first assertions que reduzem drasticamente a flakiness.

Um teste na prática

import { test, expect } from '@playwright/test';

test('usuário faz login com sucesso', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('E-mail').fill('qa@exemplo.com');
  await page.getByLabel('Senha').fill('s3nh4-segura');
  await page.getByRole('button', { name: 'Entrar' }).click();

  await expect(page.getByRole('heading', { name: 'Bem-vindo' })).toBeVisible();
});

Repare no uso de getByRole e getByLabel: localizadores baseados em acessibilidade, mais resistentes a mudanças de layout do que seletores CSS frágeis.

Boas práticas que adotei

  • Usar Page Object Model para encapsular telas e reduzir duplicação.
  • Preferir localizadores acessíveis (getByRole, getByText) a XPaths longos.
  • Rodar com --trace on-first-retry para capturar evidências só quando falha.
  • Isolar dados de teste e usar fixtures para setup/teardown.
Teste E2E bom não é o que cobre tudo — é o que falha só quando o produto realmente quebrou.

Conclusão

Playwright não substitui testes unitários nem de integração; ele coroa a pirâmide validando os fluxos críticos do ponto de vista do usuário. Combinado a um bom pipeline de CI, dá ao time a confiança de entregar mais rápido sem medo de regressões.
