Gestão de dados de teste: o gargalo invisível da automação

Diogo Gomes  ·  30 de Junho, 2026  ·  5 min de leitura

Pergunte a qualquer pessoa que mantém uma suíte de automação qual a maior causa de teste instável, e a resposta raramente é "a lógica do teste". Quase sempre é dado: o usuário que outro teste apagou, o cupom que já foi usado, o registro que existia ontem e hoje não. Escrever a asserção é a parte fácil; garantir que o sistema está no estado certo antes dela rodar é o trabalho de verdade — e o mais negligenciado.

Por que dado é tão difícil quanto a lógica

Um teste só é confiável se o estado em que ele depende for previsível. O problema é que dado tem três inimigos naturais: ele muda (alguém alterou o registro), ele some (foi apagado), e ele colide (dois testes mexem no mesmo registro ao mesmo tempo). Quando um teste verde de repente fica vermelho "sem ninguém ter mexido em nada", o culpado quase sempre é o dado — e isso é uma das maiores fontes de testes flaky.

De onde vem o dado: as quatro estratégias

Há quatro formas de colocar o sistema no estado necessário, da mais frágil à mais robusta:

EstratégiaComo éProblema / quando usar
Dado de produçãoCópia (mascarada) do banco realRealista, mas instável e arriscado (LGPD); evite depender
Seed compartilhadoMassa fixa carregada uma vez para todosCômodo, mas vira "estado global" que os testes corrompem entre si
FixtureConjunto pré-definido por teste/cenárioBom para dados de leitura estáticos
Factory (sob demanda)O teste cria o que precisa na hora, com dados únicosA mais robusta para dados que o teste modifica

A tendência moderna é clara: menos seed global, mais factory. Em vez de "esses 50 usuários existem para todo mundo", cada teste cria o seu próprio usuário no início, com um identificador único. Ninguém pisa no pé de ninguém.

A regra de ouro: cada teste cuida do próprio estado

O princípio que resolve a maior parte dos problemas: um teste deve criar o que precisa, usar, e limpar depois — sem depender de nada que outro teste tenha feito. Isso garante a independência que permite rodar os testes em qualquer ordem e em paralelo.

// Padrão arrange → act → assert → cleanup
beforeEach: usuario = criarUsuario({ email: `teste+${uuid()}@ex.com` })
test:       login(usuario); comprar(produto)
            expect(pedido).toBe("confirmado")
afterEach:  removerUsuario(usuario)   // não deixa lixo para trás

O uuid() no e-mail é o detalhe que evita colisão: mesmo rodando 10 cópias do teste em paralelo, cada uma tem seu usuário único. E o cleanup no afterEach impede que a falha de hoje vire o dado órfão que quebra o teste de amanhã.

Um teste que depende de um dado que ele não criou é uma bomba-relógio. Funciona até o dia em que aquele registro muda — e aí você perde uma tarde caçando um "bug" que nunca esteve no código.

Criar pela API, não pela interface

Para preparar o estado, prefira a rota mais barata e estável. Se o seu teste de checkout precisa de um usuário com produtos no carrinho, não automatize o cadastro e a navegação inteira pela tela só para chegar lá — crie o usuário e o carrinho direto por chamadas de API (ou direto no banco) no setup, e reserve a interface para o que você de fato quer testar. É mais rápido, e a preparação não quebra quando a tela de cadastro muda. Setup pesado pela UI é uma das receitas mais comuns de suíte lenta e frágil.

Como manter o ambiente limpo

Mesmo com cada teste limpando o seu, o banco de teste acumula sujeira com o tempo. Três abordagens, da mais forte à mais prática:

  • Transação revertida: roda o teste dentro de uma transação e dá rollback no fim. Limpeza perfeita e instantânea — mas só funciona em testes que batem no mesmo banco diretamente, não atravessando a rede.
  • Reset entre execuções: recriar o schema ou recarregar um snapshot limpo antes da suíte. Garante um ponto de partida conhecido a cada rodada.
  • Cleanup explícito: o afterEach/afterAll apaga o que criou. Mais trabalhoso, mas é o que funciona em testes E2E de ponta a ponta.

Containers efêmeros (subir um banco em Docker só para a rodada do pipeline e destruir no fim) combinam o melhor dos mundos: ambiente isolado, limpo por construção, descartado ao terminar.

Conclusão

Dado de teste é o gargalo invisível: ele não aparece na asserção, mas é onde a maioria das suítes apodrece. Prefira factories sob demanda a seed global, faça cada teste criar e limpar o próprio estado com identificadores únicos, prepare o cenário pela API em vez da tela, e mantenha o ambiente limpo com transação, reset ou containers efêmeros. Resolva o dado e você elimina a maior fatia dos testes flaky de uma vez.
