Test doubles: mock, stub, spy, fake e dummy

Diogo Gomes  ·  19 de Junho, 2026  ·  7 min de leitura

"Qual a diferença entre mock e stub?" é uma das perguntas técnicas que mais separa quem decora de quem entende. Na prática do dia a dia, todo teste de unidade isola a peça sob teste das suas dependências — banco, API externa, relógio, e-mail — e o faz com test doubles (dublês de teste). O termo guarda-chuva é de Gerard Meszaros, e existem cinco tipos, do mais bobo ao mais esperto.

O espectro dos dublês

  só ocupa espaço ──────────────────────► verifica interação
   DUMMY → STUB → FAKE → SPY → MOCK

   DUMMY  passado mas nunca usado (preenche um parâmetro)
   STUB   devolve respostas prontas ("quando perguntarem X, responda Y")
   FAKE   implementação real, porém simplificada (ex.: banco em memória)
   SPY    como o stub, mas grava como foi chamado (para checar depois)
   MOCK   já vem com a expectativa do que DEVE ser chamado (falha se não for)

Dummy — só para preencher

Um objeto passado apenas para satisfazer a assinatura, nunca de fato usado no caminho testado. Ex.: um logger exigido pelo construtor, mas irrelevante para o que estou verificando.

const loggerDummy = {};
const servico = new Servico(loggerDummy); // só para construir

Stub — respostas prontas

O stub controla a entrada indireta: ele força a dependência a devolver um valor fixo, para eu testar como meu código reage. Não verifico como foi chamado — só uso a resposta.

// Quero testar a regra de frete grátis sem chamar a API real de CEP
const apiCepStub = {
  consultar: () => ({ regiao: 'sudeste', taxa: 0 })
};

const total = calcularEntrega(apiCepStub, carrinho);
expect(total.freteGratis).toBe(true); // verifico ESTADO

Fake — implementação real, porém leve

O fake tem lógica de verdade, só que simplificada e inadequada para produção. O exemplo clássico é um repositório que guarda dados num array em memória em vez de bater no banco — rápido e sem efeito colateral.

class RepositorioFake {
  constructor() { this.itens = []; }
  salvar(item) { this.itens.push(item); }
  buscarPorId(id) { return this.itens.find(i => i.id === id); }
}

Spy — grava o que aconteceu

O spy embrulha uma chamada e registra como ela foi usada — quantas vezes, com quais argumentos — para eu inspecionar depois da ação. É o meio-termo: não impõe expectativa antes, mas permite verificar comportamento.

const emailSpy = jest.fn();
notificarUsuario(usuario, emailSpy);

// verifico COMPORTAMENTO: foi chamado certo?
expect(emailSpy).toHaveBeenCalledTimes(1);
expect(emailSpy).toHaveBeenCalledWith('ana@exemplo.com');

Mock — expectativa embutida

O mock é o mais opinativo: você define antes o que ele espera receber, e o próprio mock falha o teste se a interação esperada não acontecer. Ele verifica comportamento por construção.

const gateway = mock(PagamentoGateway);
when(gateway.cobrar(100)).thenReturn({ ok: true });

finalizarCompra(gateway, 100);

// o mock exige que cobrar(100) tenha sido chamado exatamente assim
verify(gateway).cobrar(100);

A distinção que importa: estado vs comportamento

Mais útil que decorar os cinco nomes é entender a divisão que eles revelam:

Verifica…Dublê típicoPergunta do teste
Estado (resultado)Stub, Fake"Dado este retorno, meu código produziu a saída certa?"
Comportamento (interação)Spy, Mock"Meu código chamou a dependência do jeito esperado?"
Stub responde, mock cobra. Stub está a serviço de uma verificação de estado; mock é a verificação — de comportamento. Confundir os dois leva a testes que verificam a coisa errada.

Cuidado com o excesso de mocks

Dublês demais acoplam o teste à implementação: você passa a testar "o método X chamou o método Y" em vez do comportamento observável. Aí qualquer refatoração quebra os testes mesmo sem bug. A regra prática é dublar o que é lento, instável ou tem efeito colateral (rede, banco, hora, e-mail) e deixar a lógica de negócio rodar de verdade.

Conclusão

Dummy preenche, stub responde, fake implementa leve, spy grava e mock cobra. Saber qual usar não é trivia — é o que mantém seus testes unitários rápidos, isolados e focados no que realmente importa: o comportamento do código, não os detalhes de quem ele chama.
