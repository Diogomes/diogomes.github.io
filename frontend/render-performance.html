Renderização e performance: reflow, repaint e o caminho crítico

Diogo Gomes  ·  13 de Junho, 2026  ·  3 min de leitura

"Por que essa página trava quando rolo a lista?" Quase sempre a resposta está em como o navegador transforma seu HTML e CSS em pixels — o caminho crítico de renderização. Entender reflow e repaint é o que separa quem só faz a tela aparecer de quem faz a tela aparecer rápido.

O caminho crítico de renderização

Para desenhar um pixel, o navegador segue uma esteira de etapas:

HTML  → DOM ┐
            ├→ Render Tree → Layout → Paint → Composite
CSS   → CSSOM┘

O DOM (estrutura) e o CSSOM (estilos) se combinam na render tree (o que é visível e como). O Layout calcula posição e tamanho de cada elemento, o Paint preenche os pixels (cor, texto, sombra) e o Composite junta as camadas na tela.

Reflow vs repaint

Quando algo muda, o navegador não refaz tudo. Existem dois níveis de retrabalho:

  • Reflow (layout): recalcular geometria — posição e tamanho. Caro, porque pode afetar elementos vizinhos e até a página inteira.
  • Repaint: redesenhar pixels sem mexer na geometria — mudar uma cor de fundo, por exemplo. Mais barato que reflow.

Analogia: reflow é remontar a planta da casa (mexe nas paredes); repaint é só repintar uma parede que já existe.

O que dispara reflow

Mudar dimensões, posição, fonte, adicionar ou remover elementos, redimensionar a janela — tudo força reflow. E há um detalhe traiçoeiro: ler certas propriedades (como offsetHeight, getBoundingClientRect) força o navegador a fazer um reflow na hora para te dar o valor atualizado.

Layout thrashing: o vilão silencioso

O layout thrashing acontece quando você intercala leituras e escritas num loop, forçando reflow repetido. O navegador queria agrupar as mudanças, mas cada leitura o obriga a recalcular imediatamente.

// RUIM: lê e escreve alternado → reflow a cada volta
for (const el of itens) {
  el.style.height = el.offsetHeight + 10 + 'px';
}

// BOM: lê tudo, depois escreve tudo
const alturas = itens.map(el => el.offsetHeight);
itens.forEach((el, i) => el.style.height = alturas[i] + 10 + 'px');

O que fazer na prática

  • Agrupe leituras e escritas: leia todas as medidas, depois aplique todas as mudanças.
  • Anime com transform e opacity: elas rodam no composite, sem reflow nem repaint.
  • Evite forçar layout síncrono: não leia offsetTop logo após escrever um estilo.
  • Virtualize listas longas: renderize só o que está visível (windowing) em vez de 10.000 linhas.
  • Use will-change com parcimônia para promover elementos animados a camadas próprias.
A renderização rápida raramente vem de "deixar o JS mais rápido" — vem de pedir menos trabalho de layout ao navegador.

É comum em entrevista pedir para você explicar a diferença entre reflow e repaint, ou por que animar transform é melhor que animar top. No dia a dia, isso aparece quando uma lista grande engasga ao rolar ou uma animação fica travada.

Conclusão

O navegador segue DOM → CSSOM → layout → paint → composite, e cada mudança pode custar um reflow (caro) ou um repaint (mais barato). Agrupe leituras e escritas, anime no composite e virtualize listas. Esse cuidado local é a base para a visão mais ampla de Core Web Vitals.
