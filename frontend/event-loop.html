O event loop: como o JavaScript faz async sem travar

Diogo Gomes  ·  17 de Junho, 2026  ·  3 min de leitura

"JavaScript é single-thread" e ao mesmo tempo "JavaScript não trava ao baixar dados". Como as duas coisas são verdade? A resposta é o event loop, talvez o conceito que mais separa o júnior do pleno — porque explica de uma vez a ordem de execução, o async e por que sua UI continua respondendo.

Single-thread, mas não-bloqueante

O JS executa uma instrução por vez, numa única thread. Se fosse só isso, baixar um arquivo de 3 segundos congelaria a página inteira. O truque é que o JS não faz esse trabalho pesado sozinho: ele delega para o ambiente (o navegador) e segue em frente. Quando o trabalho termina, o resultado volta para a fila.

A call stack

A call stack é a pilha de funções em execução. Chamou uma função, empilhou; ela retornou, desempilhou. É aqui que o "single-thread" vive: só existe uma pilha, e ela processa uma coisa de cada vez.

Web APIs: o trabalho terceirizado

Quando você chama setTimeout, fetch ou escuta um clique, quem cuida disso são as Web APIs do navegador — fora da thread do JS. Pense num restaurante: o garçom (a thread) anota o pedido e entrega na cozinha (Web API) em vez de ficar parado cozinhando. Ele volta a atender outras mesas.

Duas filas: macrotasks e microtasks

Quando a Web API termina, o callback não roda na hora — ele entra numa fila. E existem duas:

  • Task queue (macrotasks): setTimeout, eventos de I/O, etc.
  • Microtask queue: .then de Promises, queueMicrotask.

A regra crucial: as microtasks têm prioridade. Após cada tarefa, o event loop esvazia toda a fila de microtasks antes de pegar a próxima macrotask.

O exemplo clássico de ordem

Esse trecho é praticamente um clássico de entrevista — saber prever a saída prova que você entende o loop.

console.log('1');
setTimeout(() => console.log('2'), 0);
Promise.resolve().then(() => console.log('3'));
console.log('4');

// Saída: 1, 4, 3, 2

Por quê? 1 e 4 são síncronos (rodam direto na stack). O setTimeout vira macrotask; o .then vira microtask. Quando a stack esvazia, o loop roda todas as microtasks (3) antes de tocar na macrotask (2).

O event loop não acelera nada — ele apenas garante que a thread nunca fique parada esperando, mantendo a UI viva enquanto o trabalho pesado acontece em paralelo.

No dia a dia isso aparece quando você precisa entender por que um await "espera" sem travar a tela, ou por que um loop pesado e síncrono realmente congela tudo — ele nunca solta a stack para o loop respirar.

Conclusão

A stack roda o código síncrono, as Web APIs cuidam do assíncrono, e o event loop reabastece a stack das filas — microtasks primeiro. Entender essa coreografia é o que permite escrever código async sem surpresas. Quando o async já é trivial, o próximo desafio é gerenciar o estado dos componentes.
