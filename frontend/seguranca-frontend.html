Segurança no front-end: XSS, CSRF e CORS sem mistério

Diogo Gomes  ·  6 de Junho, 2026  ·  3 min de leitura

Segurança costuma ser tratada como "problema do back-end", mas o navegador é a linha de frente e o front-end carrega responsabilidades reais. XSS, CSRF e CORS são os três temas que mais geram confusão — e desmistificá-los, sabendo o que é de fato sua responsabilidade, é o que se espera de um sênior.

XSS: código injetado na sua página

O Cross-Site Scripting acontece quando um atacante consegue injetar JavaScript que roda no navegador de outros usuários — roubando cookies, sessão, ou agindo como a vítima. Há três sabores:

  • Stored: o script malicioso é salvo no servidor (ex.: num comentário) e servido a todos.
  • Reflected: o script vem na URL/requisição e é refletido de volta na resposta.
  • DOM-based: o problema é puramente no JS do cliente, que joga dado não confiável no DOM.

Defendendo de XSS

A regra de ouro: nunca confie em entrada e sempre escape a saída conforme o contexto. Frameworks modernos escapam por padrão ao renderizar texto — o perigo mora nas brechas como innerHTML ou dangerouslySetInnerHTML.

// PERIGOSO: interpreta HTML do usuário
el.innerHTML = comentarioDoUsuario;

// SEGURO: trata como texto puro
el.textContent = comentarioDoUsuario;

Como segunda camada, o CSP (Content Security Policy) é um header que diz ao navegador de onde scripts podem vir, bloqueando código injetado mesmo que ele passe. Defesa em profundidade.

CSRF: usar sua sessão sem você saber

O Cross-Site Request Forgery explora o fato de o navegador enviar seus cookies automaticamente. Um site malicioso faz seu navegador disparar uma requisição para um site onde você está logado — e o servidor, vendo o cookie válido, executa a ação como se fosse você (ex.: transferir dinheiro).

Defendendo de CSRF

  • Token anti-CSRF: um valor imprevisível que só sua página conhece e envia junto; o site atacante não consegue adivinhá-lo.
  • Cookie SameSite: instrui o navegador a não enviar o cookie em requisições vindas de outros sites.
Set-Cookie: sessao=abc; SameSite=Strict; Secure; HttpOnly

CORS: o mal-entendido número um

Talvez nenhum tema gere mais confusão. O CORS não é uma proteção do seu servidor que você "desliga". É uma política do navegador: por padrão (same-origin policy), o JS de um site não pode ler respostas de outra origem. O CORS é o mecanismo pelo qual um servidor autoriza origens específicas a lerem suas respostas.

Para requisições "não simples", o navegador faz antes um preflight: um OPTIONS perguntando "posso?". Se o servidor responde com os headers certos, a requisição real prossegue.

O equívoco clássico: "desabilitar o CORS para resolver o erro". Você não desabilita o CORS — você configura o servidor para permitir as origens certas. Pôr Access-Control-Allow-Origin: * em uma API com credenciais é abrir a porta, não consertar a fechadura.

É importante entender: CORS protege o usuário (impede que um site leia dados de outro em nome dele), não protege sua API contra acesso direto. Quem chama sua API por fora do navegador (curl, outro back-end) ignora CORS completamente — a autorização real é autenticação no servidor.

O que é responsabilidade do front

  • Tratar toda entrada/saída como suspeita: escapar antes de inserir no DOM.
  • Evitar innerHTML com dados de usuário; sanitizar quando inevitável.
  • Configurar e respeitar CSP e cookies SameSite/HttpOnly.
  • Entender CORS para diagnosticar (e não "burlar") erros de origem.
  • Nunca guardar segredos no front: tudo que vai ao navegador é público.

É comum em entrevista pedir para você diferenciar XSS de CSRF ou explicar por que CORS não é uma defesa do servidor. No dia a dia, isso aparece no primeiro erro de "blocked by CORS policy" ou numa revisão de código que flagra um innerHTML perigoso.

Conclusão

XSS injeta script (escape a saída e use CSP), CSRF abusa da sessão (token e SameSite) e CORS é uma política do navegador para autorizar leitura entre origens — não um interruptor de segurança. Segurança no front é, acima de tudo, desconfiar de todo dado e entender de quem é cada responsabilidade.
