Estado em front-end: props, state e quando subir o estado

Diogo Gomes  ·  15 de Junho, 2026  ·  3 min de leitura

Frameworks de componentes (React, Vue, Svelte) giram em torno de uma pergunta: onde mora o dado e quem pode mudá-lo? Acertar isso é a diferença entre uma UI previsível e um emaranhado de bugs onde nada está sincronizado. Estado e props são os dois lados dessa moeda.

Props: a entrada imutável

Props são os dados que um componente recebe de quem o usa. Da perspectiva do componente, são imutáveis: ele lê, mas não altera. Pense numa receita de bolo — os ingredientes vêm de fora, e a receita não pode reescrever a lista de compras; só usar o que chegou.

State: o dado interno mutável

State é o dado que o componente possui e pode mudar ao longo do tempo: o texto de um input, se um modal está aberto, um contador. Quando o state muda, o framework re-renderiza o componente para refletir o novo valor.

function Contador() {
  const [n, setN] = useState(0);
  return (
    <button onClick={() => setN(n + 1)}>
      Cliquei {n}x
    </button>
  );
}

Fluxo de dados unidirecional

Os dados descem de pai para filho via props; os eventos sobem via callbacks. Esse fluxo unidirecional é o que torna a UI previsível: para saber por que algo está na tela, você segue o dado de cima para baixo, numa única direção. É como uma cachoeira — a água só desce.

Lifting state up: subir o estado

E quando dois componentes irmãos precisam do mesmo dado? Você não duplica o state em cada um — isso gera dessincronização. Você sobe o estado (lifting state up) para o ancestral comum mais próximo, que então passa o valor para baixo via props. O dado vive num único lugar: a "fonte da verdade".

function Pai() {
  const [texto, setTexto] = useState('');
  return (
    <>
      <Input valor={texto} aoMudar={setTexto} />
      <Preview valor={texto} />
    </>
  );
}

Estado derivado e inputs controlados

Cuidado com estado derivado: se um valor pode ser calculado a partir de outro state, não o guarde separadamente — calcule na hora. Guardar a lista e também "quantos itens" cria duas fontes da verdade que vão divergir.

Um input controlado é aquele cujo valor vem do state: o que aparece na tela é sempre o que está no state, e digitar dispara um evento que atualiza o state. Isso te dá controle total — validar, formatar, limpar — porque o React é dono do valor, não o DOM.

A regra que evita 90% dos bugs de estado: cada dado tem uma única fonte da verdade. Se o mesmo valor existe em dois lugares, eles vão se desencontrar.

Quando partir para estado global

Subir o estado funciona até você se pegar passando a mesma prop por cinco níveis ("prop drilling") só para chegar lá embaixo. Aí entra estado global (Context, Redux, Zustand, signals): dados que muitas partes distantes da árvore consomem — usuário logado, tema, carrinho. Mas não comece por aí: estado global tem custo de complexidade. Use-o quando o local realmente não escala.

É comum em entrevista pedir para explicar a diferença entre props e state, ou descrever quando você faria lifting state up. No dia a dia, isso aparece toda vez que dois componentes "deveriam mostrar o mesmo número" e não mostram.

Conclusão

Props entram e não mudam; state é interno e muda; dados descem e eventos sobem. Suba o estado quando irmãos compartilham um dado, evite duplicar o que é derivável e só recorra ao global quando o prop drilling doer. Com a UI previsível, o próximo foco é fazê-la renderizar rápido.
