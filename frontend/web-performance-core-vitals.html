Web performance: Core Web Vitals, lazy loading e bundle

Diogo Gomes  ·  8 de Junho, 2026  ·  3 min de leitura

Performance deixou de ser "achismo" e virou métrica medível e até fator de ranqueamento no Google. Os Core Web Vitals são o padrão da indústria para responder "esse site é bom de usar?". Saber o que cada um mede, por que piora e como corrigir é parte central do trabalho de sênior.

As três métricas

Os Core Web Vitals atuais medem três experiências distintas: o quão rápido o conteúdo principal aparece, o quão estável é o layout e o quão responsivo é o site à interação.

MétricaO que medeAlvo "bom"
LCPTempo até o maior elemento visível pintar≤ 2,5 s
CLSQuanto o layout "pula" sem aviso≤ 0,1
INPLatência da resposta às interações≤ 200 ms

LCP: o conteúdo principal aparece rápido?

O Largest Contentful Paint marca quando o maior elemento (geralmente a imagem do herói ou um bloco de texto) é renderizado. Causas comuns de LCP ruim: servidor lento, CSS/JS bloqueando a renderização, imagens enormes. Correções: priorizar o recurso crítico (fetchpriority="high"), usar CDN, comprimir imagens e evitar fontes que travam o texto.

CLS: o layout fica parado?

O Cumulative Layout Shift mede o pulo visual — aquele momento em que você vai clicar num botão e um anúncio empurra tudo para baixo. As causas clássicas: imagens sem dimensões reservadas, fontes que trocam de tamanho ao carregar, conteúdo injetado acima do que já estava lá. Correção: sempre reserve espaço (width/height ou aspect-ratio nas imagens).

<!-- Reserva espaço → não pula -->
<img src="foto.jpg" width="800" height="600" alt="...">

INP: o site responde ao toque?

O Interaction to Next Paint mede quanto tempo entre o clique e a tela responder. INP ruim quase sempre é JavaScript pesado bloqueando a thread principal — o que conecta direto com a renderização e o caminho crítico. Correção: quebrar tarefas longas, adiar trabalho não-urgente e evitar handlers caros no clique.

Carregar menos: splitting, lazy e imagens

A maior alavanca de performance é simplesmente enviar menos código e bytes:

  • Code splitting: dividir o bundle por rota, carregando só o necessário para a tela atual.
  • Lazy loading: adiar o que está fora da tela — loading="lazy" em imagens, import() dinâmico para componentes.
  • Imagens: formatos modernos (WebP/AVIF), tamanhos responsivos (srcset), compressão.
// Carrega o componente só quando for usado
const Grafico = lazy(() => import('./Grafico'));

Análise de bundle e performance percebida

Um bundle analyzer mostra o que está engordando seu pacote — muitas vezes uma biblioteca pesada importada inteira por um único uso. Cortar isso costuma render mais que qualquer micro-otimização.

E há a performance percebida: o usuário sente velocidade, não mede milissegundos. Skeletons em vez de spinners, mostrar conteúdo parcial, feedback instantâneo no clique — tudo faz o site parecer mais rápido mesmo quando os números são parecidos.

A otimização que mais rende quase nunca é deixar o código mais esperto — é não enviar o código que o usuário não precisa agora.

É comum em entrevista de sênior pedir para você citar os Core Web Vitals, seus limiares e como atacaria um LCP ruim. No dia a dia, isso aparece quando o Lighthouse acende vermelho ou o produto reclama que "o site demora a aparecer".

Conclusão

LCP cuida do conteúdo aparecer, CLS da estabilidade visual e INP da responsividade. As alavancas são quase sempre as mesmas: enviar menos (splitting, lazy, imagens otimizadas) e cuidar da percepção. Performance é experiência — e, com ela e a arquitetura sob controle, falta proteger o usuário com segurança.
