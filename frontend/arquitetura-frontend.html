Arquitetura de front-end: componentização, design system e monorepo

Diogo Gomes  ·  10 de Junho, 2026  ·  3 min de leitura

Num app pequeno, qualquer estrutura funciona. O problema aparece quando o time cresce, as telas se multiplicam e cada mudança parece quebrar três coisas distantes. Arquitetura de front-end é o conjunto de decisões que mantém um app grande compreensível, consistente e barato de evoluir — e é o que se espera de um sênior conduzir.

Componentização: do átomo à página

A base de tudo é quebrar a UI em peças reutilizáveis. O atomic design dá um vocabulário útil: átomos (botão, input), moléculas (campo de busca = input + botão), organismos (um cabeçalho completo), templates e páginas. Não é dogma — é uma forma de pensar composição: peças pequenas e burras embaixo, peças que orquestram em cima.

Um bom componente faz uma coisa e não sabe onde será usado. No momento em que ele "conhece" a tela que o contém, virou acoplamento disfarçado de reúso.

Design system e design tokens

Um design system é a biblioteca compartilhada de componentes + regras de uso que garante que botão, cor e espaçamento sejam iguais em todo o produto. No coração dele estão os design tokens: variáveis nomeadas para valores de design (--color-primary, --space-md). Trocar a marca inteira vira mudar tokens, não caçar hex codes pelo código.

:root {
  --color-primary: #4f46e5;
  --space-md: 16px;
}
.btn { background: var(--color-primary); padding: var(--space-md); }

Organização por feature, não por tipo

Muitos projetos colapsam organizando por tipo de arquivo (uma pasta gigante de componentes, outra de hooks, outra de utils). Em escala, organize por feature: tudo que pertence ao "checkout" — componentes, estado, chamadas de API, testes — vive junto. Assim você lê uma funcionalidade num lugar só, e deletá-la é apagar uma pasta.

src/
  features/
    checkout/   (componentes, hooks, api, testes)
    perfil/
  shared/       (design system, utils genéricos)

Escolha da gerência de estado

Não existe "a melhor" — existe a adequada à natureza do dado. Estado de servidor (dados de API) pede ferramentas de cache como React Query. Estado de UI local fica no componente. Estado global de cliente (tema, sessão) pede Context ou um store leve. O erro comum do sênior iniciante é jogar tudo num Redux gigante. Separe por tipo de estado e a complexidade despenca.

Fronteiras de módulo

O que mantém um app grande sadio são fronteiras claras: cada feature expõe uma API pública (um index) e esconde o resto. Outras features importam só o que foi exposto. Como apartamentos num prédio — você usa o corredor comum, não atravessa a sala do vizinho. Isso permite refatorar o interior de um módulo sem quebrar o mundo.

O trade-off do monorepo

Monorepo é manter vários pacotes/apps num único repositório. Tem vantagens reais, mas custos reais também:

VantagensCustos
Código compartilhado fácilTooling de build mais complexo
Refatoração atômica (um PR)CI fica lento sem cache esperto
Versão única, sem driftRepositório grande, mais governança

A decisão é contextual: times pequenos ganham pouco e pagam o tooling; organizações com muito código compartilhado ganham muito. É comum em entrevista de sênior pedir justamente para você defender essa escolha — e o esperado não é a resposta "certa", é o raciocínio sobre trade-offs.

Como estruturar um app grande, em resumo

  • Componentes pequenos e desacoplados, compostos de baixo para cima.
  • Design system + tokens como única fonte de verdade visual.
  • Organize por feature, com fronteiras de módulo explícitas.
  • Estado segmentado por tipo (servidor, UI, global).
  • Monorepo só quando o ganho de compartilhamento supera o custo de tooling.

Conclusão

Arquitetura de front-end é gerir acoplamento: peças pequenas, fronteiras claras, estado segmentado e visual centralizado em tokens. Decisões como monorepo são trade-offs, não verdades. Com a estrutura sólida, a próxima conversa de sênior é entregar isso rápido para o usuário.
