CI/CD na prática: do commit ao deploy automático

Diogo Gomes  ·  20 de Junho, 2026  ·  3 min de leitura

Antes da automação, soltar uma nova versão de software era um evento tenso: juntava-se semanas de código de várias pessoas, rezava-se para tudo funcionar e o deploy era feito à mão na madrugada. CI/CD inverteu essa lógica. A ideia é integrar e entregar com frequência, em pequenos passos automatizados, para que o software esteja sempre pronto e os erros apareçam cedo, quando ainda são baratos de corrigir.

CI: integração contínua

Continuous Integration significa integrar o código de todo mundo com frequência — várias vezes ao dia — e, a cada integração, rodar automaticamente o build e os testes. Assim, se alguém quebra algo, o time descobre em minutos, não dias depois. Isso depende de um bom uso de branches e Pull Requests.

CD: delivery vs deployment

O "CD" tem dois significados que confundem muita gente, e é comum em entrevista pedir para distinguir os dois:

  • Continuous Delivery — toda mudança aprovada fica pronta para ir a produção a qualquer momento, mas o último passo (apertar o botão) é manual.
  • Continuous Deployment — vai mais longe: cada mudança que passa nos testes vai a produção automaticamente, sem intervenção humana.

Os estágios de um pipeline

Um pipeline é uma sequência de etapas que o código atravessa, do commit até a produção. Cada etapa só roda se a anterior passou. Um fluxo típico:

commit → build → testes → empacota (artefato) → deploy staging → deploy produção

Se os testes falham, o pipeline para ali e ninguém precisa investigar um deploy quebrado depois.

Artefatos

O artefato é o resultado empacotado do build — um arquivo .jar, um binário ou, cada vez mais comum, uma imagem de container. A regra de ouro é: construa o artefato uma vez e promova o mesmo artefato por todos os ambientes. Assim você garante que o que foi testado em staging é exatamente o que vai a produção.

O valor real do CI/CD não é "fazer deploy rápido": é encurtar o ciclo de feedback. Quanto antes você descobre que algo quebrou, menor o estrago e mais barato o conserto.

Exemplo de pipeline em YAML

A maioria das ferramentas (GitHub Actions, GitLab CI, etc.) descreve o pipeline num arquivo YAML versionado junto ao código:

jobs:
  build-e-teste:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
      - run: npm run build

Por que isso importa

Os ganhos práticos são diretos: feedback rápido (erros aparecem em minutos), menos regressão (testes automáticos pegam o que o humano esqueceria) e deploys previsíveis (o processo é o mesmo toda vez, sem passos manuais que alguém pode errar). Times com bom CI/CD entregam mais e com menos medo.

Conclusão

CI/CD é a espinha dorsal do DevOps moderno: transforma entregas de eventos arriscados em rotina tranquila. Comece entendendo a diferença entre integração e entrega, o que é um artefato e por que o feedback rápido muda tudo. Com isso, você já consegue ler e contribuir com a maioria dos pipelines que vai encontrar.
