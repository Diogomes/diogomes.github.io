Infraestrutura como código: Terraform e o fim do clicar no console

Diogo Gomes  ·  15 de Junho, 2026  ·  3 min de leitura

Houve um tempo em que montar infraestrutura significava abrir o console da nuvem e clicar: criar uma máquina aqui, um banco ali, ajustar uma regra de rede acolá. Funciona para uma coisa só. Mas quando você precisa recriar tudo idêntico em outra região, ou descobrir quem mudou o quê três meses atrás, o clique vira pesadelo. Infraestrutura como código (IaC) resolve isso tratando servidores, redes e bancos como código versionado.

O que é infraestrutura como código

IaC é descrever sua infraestrutura em arquivos de texto, versionados no Git como qualquer outro código. Em vez de clicar para criar um servidor, você escreve a definição dele e uma ferramenta a aplica. A infraestrutura passa a ser revisável, reproduzível e rastreável.

Declarativo vs imperativo

Essa distinção cai bastante em entrevista. Na abordagem imperativa, você descreve os passos ("crie esta VM, depois instale isto, depois abra esta porta"). Na declarativa, você descreve o resultado desejado ("quero 3 servidores com esta configuração") e a ferramenta descobre como chegar lá. Terraform é declarativo.

AbordagemVocê descreveExemplo
ImperativoO passo a passoScripts bash, Ansible (em parte)
DeclarativoO estado final desejadoTerraform, CloudFormation

O state file

O Terraform mantém um state file: um registro do que ele já criou no mundo real. É assim que ele compara o que você declarou com o que existe e decide o que mudar. Esse arquivo é crítico — costuma ser guardado em um local remoto e compartilhado (ex. um bucket) para que o time trabalhe sobre a mesma verdade, e protegido porque pode conter dados sensíveis.

O ciclo plan / apply

O fluxo de trabalho do Terraform é desenhado para evitar surpresas. Primeiro você roda plan, que mostra exatamente o que vai mudar, sem mexer em nada. Você revisa, e só então roda apply.

terraform plan    # mostra: +2 para criar, ~1 alterar, -0 destruir
terraform apply   # aplica as mudanças após você confirmar
O plan é o seu cinto de segurança. Ele transforma "vamos torcer para dar certo" em "eu vi exatamente o que vai acontecer antes de acontecer". Nunca aplique em produção sem ler o plan.

Idempotência

Uma propriedade central da IaC declarativa é a idempotência: aplicar a mesma configuração várias vezes leva sempre ao mesmo estado. Se a infraestrutura já está como você declarou, o apply não faz nada. Isso é o oposto de um script que, rodado duas vezes, cria recursos duplicados.

Exemplo em HCL

Terraform usa a linguagem HCL. Definir um recurso é direto e legível:

resource "aws_instance" "web" {
  ami           = "ami-0abc123"
  instance_type = "t3.micro"
  tags = {
    Name = "servidor-web"
    Env  = "producao"
  }
}

Por que clicar no console não escala

No dia a dia, IaC aparece quando você precisa de ambientes idênticos (dev, staging, produção), recuperar-se de um desastre recriando tudo, ou auditar mudanças. Clicar no console não deixa histórico, não é revisável por um colega num Pull Request e é impossível de reproduzir com fidelidade. Código resolve os três problemas de uma vez — e se conecta naturalmente ao pipeline de CI/CD, que pode aplicar a infraestrutura automaticamente.

Conclusão

Infraestrutura como código traz para o mundo dos servidores as mesmas garantias que o Git trouxe para o software: versionamento, revisão e reprodutibilidade. Entender o modelo declarativo, o papel do state file e o ciclo plan/apply é o que permite gerenciar dezenas de ambientes sem caos — e o que esperam de um DevOps pleno.
