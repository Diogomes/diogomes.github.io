Git além do commit: branches, merge e fluxos de trabalho

Diogo Gomes  ·  22 de Junho, 2026  ·  3 min de leitura

Saber dar git commit e git push é o começo, mas o trabalho real começa quando várias pessoas mexem no mesmo código ao mesmo tempo. É aí que branches, merge, rebase e fluxos de trabalho entram em cena. Entender essas peças evita conflitos dolorosos e mantém o histórico do projeto limpo e compreensível.

Commit e branch: a base

Um commit é uma fotografia do código em um momento. Uma branch é uma linha de desenvolvimento independente — uma forma de trabalhar numa funcionalidade sem afetar o código estável. Criar uma branch é barato e rápido, e essa é justamente a graça do Git.

git switch -c feature/login   # cria e entra na branch
git add .
git commit -m "adiciona tela de login"

Merge vs rebase

Os dois servem para juntar o trabalho de uma branch na outra, mas de formas diferentes. O merge cria um commit de junção e preserva o histórico real. O rebase reescreve os commits da sua branch como se tivessem nascido a partir do topo da outra, gerando um histórico linear. É comum em entrevista pedir para explicar a diferença e quando usar cada um.

AspectoMergeRebase
HistóricoPreserva (com commit de merge)Reescreve (linear)
RastreabilidadeMostra quando algo foi integradoHistórico mais limpo, perde o "quando"
RiscoSeguro em branches compartilhadasPerigoso em branches já publicadas
Uso típicoTrazer feature para a mainAtualizar sua branch local antes do PR
A regra de ouro: nunca faça rebase de uma branch que outras pessoas já usam. Rebase reescreve a história, e reescrever história compartilhada quebra o trabalho dos colegas.

O que é um conflito

Um conflito acontece quando duas branches alteraram a mesma linha de um arquivo de formas diferentes. O Git não sabe qual versão é a certa e pede que você decida. Não é um erro — é uma situação normal que você resolve editando o arquivo e marcando como resolvido.

<<<<<<< HEAD
porta = 8080
=======
porta = 3000
>>>>>>> feature/config
# escolha uma versão, apague os marcadores, git add e commit

Trabalhando com o remoto

O repositório remoto (ex. GitHub) é a cópia compartilhada. Três comandos formam o vocabulário básico:

  • push — envia seus commits para o remoto.
  • pull — traz e já junta as mudanças do remoto.
  • fetch — traz as mudanças sem juntar (você decide depois).

Diagrama de branches

main    A---B---C-------F   (F = merge)
                 \     /
feature           D---E

A branch feature nasceu de B, recebeu os commits D e E, e foi integrada em F via merge.

Fluxos de trabalho

Não basta saber os comandos: equipes adotam um fluxo combinado. Os três mais comuns:

  • Feature branch + Pull Request — cada tarefa em sua branch, revisada via PR antes de entrar na main. O padrão da maioria dos times.
  • Trunk-based — todos integram com frequência numa única branch principal, branches vivem horas, não dias. Favorece CI/CD contínuo.
  • GitFlow — branches separadas para develop, release e hotfix. Robusto, porém pesado; bom para releases versionadas.

Conclusão

Git é a memória do projeto e a forma como o time colabora. Entender merge vs rebase, saber resolver conflitos sem pânico e conhecer os fluxos de trabalho são habilidades que você usa todos os dias. Um histórico limpo e um fluxo bem definido tornam o trabalho de todos — incluindo o pipeline de CI/CD — muito mais previsível.
