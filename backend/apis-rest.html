APIs REST: o que torna uma API RESTful

Diogo Gomes  ·  22 de Junho, 2026  ·  3 min de leitura

REST é o estilo de arquitetura mais comum para construir APIs web hoje. Mas "ter endpoints HTTP" não torna uma API RESTful — REST é um conjunto de princípios sobre como modelar e expor recursos. Entender esses princípios é o que separa uma API previsível e fácil de consumir de uma bagunça.

A API como contrato

Uma API é um contrato entre quem fornece e quem consome. Ela define quais operações existem, o que cada uma espera receber e o que devolve. Se o contrato é claro e estável, times diferentes (ou empresas diferentes) conseguem trabalhar sem combinar nada além da documentação. Esse contrato roda sobre HTTP — vale revisar como a web funciona antes de seguir.

Recursos são substantivos

O princípio central de REST é pensar em recursos: coisas do seu domínio, expressas como substantivos no plural. /produtos, /usuarios, /pedidos. O que você faz com o recurso é expresso pelo verbo HTTP, não pela URL. Por isso /criarProduto não é RESTful — o "criar" deveria ser um POST /produtos.

Verbos HTTP e a tabela de endpoints

VerboEndpointAção
GET/produtosLista todos os produtos
GET/produtos/42Busca o produto 42
POST/produtosCria um novo produto
PUT/produtos/42Substitui o produto 42 inteiro
PATCH/produtos/42Atualiza parte do produto 42
DELETE/produtos/42Remove o produto 42

Statelessness e representações

REST é stateless: cada requisição carrega tudo que o servidor precisa para processá-la. O servidor não guarda "onde você estava" entre chamadas. Isso é o que permite escalar horizontalmente (qualquer servidor atende qualquer requisição). O recurso em si é abstrato; o que trafega é uma representação dele, quase sempre em JSON:

GET /produtos/42

200 OK
{
  "id": 42,
  "nome": "Teclado mecânico",
  "preco": 350.00,
  "emEstoque": true
}

Status codes corretos

Usar o status code certo faz parte do contrato. Em entrevista é comum pedirem para explicar as famílias: 2xx sucesso (200 OK, 201 Created), 4xx erro do cliente (400 inválido, 401 não autenticado, 403 sem permissão, 404 não encontrado), 5xx erro do servidor. Retornar 200 OK com um corpo "deu erro" dentro é um clássico de API mal feita.

Métodos seguros e idempotentes

Dois conceitos importantes: um método é seguro se não altera dados (GET). É idempotente se repeti-lo produz o mesmo efeito de fazê-lo uma vez (GET, PUT, DELETE). POST não é idempotente — dois POSTs criam dois recursos. Essa noção volta a aparecer quando falamos de resiliência e retries.

O que torna uma API RESTful não é a tecnologia, e sim a disciplina: recursos como substantivos, verbos HTTP fazendo o trabalho, statelessness e status codes honestos.

O que torna uma API "não-RESTful"

Sinais de alerta: verbos na URL (/getProduto), tudo via POST, status sempre 200, ou guardar estado de sessão no servidor entre chamadas. No dia a dia isso aparece quando uma API "funciona" mas é imprevisível — cada endpoint tem sua própria lógica e você precisa ler o código para entender.

Conclusão

REST é menos sobre regras rígidas e mais sobre previsibilidade. Modele recursos, use os verbos HTTP corretamente, mantenha o servidor stateless e seja honesto nos status codes. Uma API que segue isso é fácil de consumir, documentar e escalar — e demonstra que você entende o contrato por trás da web.
