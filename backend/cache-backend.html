Cache no back-end: quando usar e como invalidar

Diogo Gomes  ·  13 de Junho, 2026  ·  3 min de leitura

Cache é uma das ferramentas mais poderosas para deixar um sistema rápido — e uma das mais fáceis de errar. A famosa frase "as duas coisas mais difíceis em computação são nomear coisas e invalidar cache" existe por um motivo. Saber quando cachear, como invalidar e quando não cachear é esperado de um dev pleno.

Por que cachear

Cache é guardar o resultado de uma operação cara para reusar depois, evitando refazê-la. A operação pode ser uma query pesada, uma chamada a outra API ou um cálculo demorado. A analogia: em vez de ir ao mercado toda vez que quer leite, você guarda na geladeira. O custo é que o leite na geladeira pode vencer — e é aí que mora toda a complexidade.

O padrão cache-aside

O padrão mais comum é o cache-aside (também chamado lazy loading): a aplicação primeiro olha o cache; se o dado estiver lá (hit), usa; se não (miss), busca na origem, guarda no cache e devolve.

function getProduto(id):
    valor = cache.get("produto:" + id)
    if valor != null:
        return valor              // cache hit
    valor = banco.query(id)       // cache miss
    cache.set("produto:" + id, valor, ttl=300)
    return valor

A vantagem: só é cacheado o que de fato é pedido. A desvantagem: o primeiro acesso a cada item é sempre lento (o miss inicial).

TTL e expiração

O TTL (Time To Live) define por quanto tempo um valor fica no cache antes de expirar automaticamente. É a forma mais simples de evitar dados eternamente desatualizados. Um TTL curto deixa os dados mais frescos mas reduz o aproveitamento; um TTL longo aproveita mais mas arrisca servir dados velhos. Escolher o TTL é equilibrar frescor contra performance.

Estratégias de escrita

O cache-aside trata da leitura. Quando os dados mudam, há estratégias de escrita. A mais direta é o write-through: ao escrever, você atualiza banco e cache juntos, mantendo-os em sincronia. Outras opções incluem invalidar a chave na escrita (deixar o próximo read recarregar). O importante é ter uma resposta para "o que acontece com o cache quando o dado muda?".

Por que invalidação é difícil

O dado em cache é uma cópia, e cópias ficam stale (desatualizadas) quando a origem muda. Você precisa decidir quando descartá-las — e errar para qualquer lado dói: invalidar demais joga fora a vantagem do cache; invalidar de menos serve dados errados. Pior: invalidar a chave certa, no momento certo, em todos os lugares onde ela existe, é genuinamente difícil em sistemas distribuídos.

Cachear é fácil; o difícil é saber a hora exata de jogar fora o dado guardado. Todo bug de "por que está mostrando o valor antigo?" é um problema de invalidação.

Onde fica o cache

Cache em memória local é o mais rápido mas não escala entre instâncias — daí o Redis ser tão usado: um cache compartilhado que todas as instâncias consultam. Isso conecta com escalabilidade, já que serviços stateless costumam delegar o estado a um cache externo.

Quando NÃO cachear

Cache não é grátis. Evite quando: os dados mudam a cada acesso (cache nunca acerta), exigem precisão absoluta (saldo bancário, estoque crítico), ou são raramente lidos (não compensa a complexidade). No dia a dia, cachear cedo demais é uma fonte comum de bugs sutis — meça primeiro, cacheie depois.

Conclusão

Cache troca frescor por velocidade. Domine o padrão cache-aside, defina TTLs conscientes, tenha uma estratégia de invalidação e saiba reconhecer os casos onde cachear traz mais problema que benefício. É uma alavanca enorme de performance — desde que você respeite a parte difícil.
