SQL essencial: CRUD, JOIN e transações

Diogo Gomes  ·  20 de Junho, 2026  ·  3 min de leitura

Quase todo sistema back-end guarda dados em um banco relacional, e a linguagem para conversar com ele é o SQL. Saber montar um CRUD, entender JOINs e ter noção de transações é conhecimento que você usa todos os dias — e que invariavelmente aparece em entrevistas técnicas.

Tabelas e linhas: o modelo relacional

Um banco relacional organiza dados em tabelas, como planilhas. Cada tabela tem colunas (campos) e linhas (registros). Uma tabela usuarios tem colunas como id, nome, email, e cada usuário é uma linha. O id costuma ser a chave primária: identifica unicamente cada linha.

CRUD: as quatro operações

CRUD significa Create, Read, Update, Delete — e cada uma mapeia para um comando SQL:

-- Create
INSERT INTO usuarios (nome, email)
VALUES ('Ana', 'ana@exemplo.com');

-- Read
SELECT id, nome FROM usuarios WHERE id = 1;

-- Update
UPDATE usuarios SET nome = 'Ana Paula' WHERE id = 1;

-- Delete
DELETE FROM usuarios WHERE id = 1;

A cláusula WHERE é crítica: ela filtra quais linhas a operação afeta. Um UPDATE ou DELETE sem WHERE atinge a tabela inteira — um erro perigoso e clássico.

JOIN: combinando tabelas

Dados relacionados ficam em tabelas separadas. Imagine usuarios e pedidos, onde cada pedido tem um usuario_id. Para listar o nome do usuário junto com seus pedidos, usamos JOIN:

SELECT u.nome, p.valor
FROM usuarios u
INNER JOIN pedidos p ON p.usuario_id = u.id;

O INNER JOIN traz só as linhas que casam dos dois lados (usuários que têm pedidos). O LEFT JOIN traz todos os usuários, mesmo sem pedidos — nesse caso as colunas de pedido vêm como NULL. Em entrevista é comum pedir para explicar a diferença:

TipoTrazUso típico
INNER JOINSó linhas que casam dos dois ladosUsuários que fizeram pedidos
LEFT JOINTodas da esquerda + casadas da direitaTodos os usuários, com ou sem pedido

Transações e ACID

Uma transação agrupa vários comandos como uma unidade: ou tudo acontece, ou nada acontece. O exemplo clássico é transferir dinheiro — debitar de uma conta e creditar em outra precisam acontecer juntos:

BEGIN;
  UPDATE contas SET saldo = saldo - 100 WHERE id = 1;
  UPDATE contas SET saldo = saldo + 100 WHERE id = 2;
COMMIT;  -- se algo falhar antes, faz ROLLBACK

As garantias de uma transação resumem-se na sigla ACID: Atomicidade (tudo ou nada), Consistência (o banco sai em estado válido), Isolamento (transações simultâneas não se atrapalham) e Durabilidade (depois do commit, os dados persistem mesmo se a energia cair).

WHERE define o alcance da sua operação e transações definem o alcance da sua segurança. Esquecer qualquer um dos dois é onde os incidentes nascem.

Índices: por que algumas queries voam

Um índice é como o índice remissivo de um livro: em vez de ler página por página, você vai direto ao ponto. Sem índice, buscar WHERE email = 'ana@exemplo.com' obriga o banco a varrer a tabela inteira (full scan). Com um índice na coluna email, a busca é quase instantânea. O custo: índices ocupam espaço e deixam escritas um pouco mais lentas, então não se indexa tudo.

Conclusão

SQL é uma habilidade que rende juros: CRUD com WHERE para o dia a dia, JOINs para conectar dados, transações para garantir integridade e índices para performance. Domine esses quatro pilares e você cobre a esmagadora maioria das tarefas de banco — e das perguntas de entrevista.
