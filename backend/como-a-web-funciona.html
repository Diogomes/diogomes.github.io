Como a web funciona: HTTP, request e response

Diogo Gomes  ·  24 de Junho, 2026  ·  3 min de leitura

Toda vez que você digita um endereço no navegador e aperta Enter, uma sequência fascinante de eventos acontece em milissegundos. Entender esse fluxo é a base de tudo no back-end: APIs, autenticação, cache e escalabilidade são apenas variações desse mesmo modelo. Se você domina request e response, domina o resto.

O modelo cliente-servidor

A web funciona com dois papéis: o cliente (seu navegador, um app, um script) faz pedidos, e o servidor responde a eles. O cliente nunca "entra" no servidor — ele apenas envia uma mensagem pedindo algo e espera uma resposta. Essa conversa acontece usando o protocolo HTTP (HyperText Transfer Protocol).

Uma boa analogia é pedir comida num restaurante. Você (cliente) faz um pedido ao garçom (a requisição). A cozinha (servidor) processa. O garçom volta com o prato ou com um "acabou esse ingrediente" (a resposta). Você não entra na cozinha: só envia pedidos e recebe respostas.

O que acontece entre a URL e a página

Ao acessar https://exemplo.com/produtos, vários passos acontecem em ordem:

1. DNS    -> "exemplo.com" vira um IP (ex: 93.184.216.34)
2. TCP    -> cliente abre uma conexao com o servidor naquele IP
3. TLS    -> handshake de criptografia (o "s" do https)
4. HTTP   -> cliente envia a requisicao: GET /produtos
5. Server -> processa (consulta banco, monta resposta)
6. HTTP   -> servidor devolve a response: 200 OK + HTML/JSON
7. Render -> navegador desenha a pagina

O DNS é a "agenda de contatos" da internet: traduz nomes legíveis em endereços IP. O TCP estabelece um canal confiável entre cliente e servidor, e o TLS o criptografa. Só então a requisição HTTP de fato viaja.

Anatomia da requisição

Uma requisição HTTP tem quatro partes. Em entrevista é comum pedir para descrevê-las:

  • Método (verbo): o que você quer fazer — GET (ler), POST (criar), PUT/PATCH (atualizar), DELETE (remover).
  • Caminho (path): qual recurso — /produtos, /usuarios/42.
  • Headers: metadados — tipo de conteúdo, autenticação, idioma.
  • Body: os dados enviados (em POST/PUT), geralmente em JSON. GET normalmente não tem body.
POST /produtos HTTP/1.1
Host: exemplo.com
Content-Type: application/json
Authorization: Bearer abc123

{ "nome": "Teclado", "preco": 350 }

Anatomia da resposta

A resposta tem três partes: o status code, os headers e o body. O status resume o que aconteceu, agrupado em famílias: 2xx sucesso, 3xx redirecionamento, 4xx erro do cliente, 5xx erro do servidor.

StatusSignificadoQuando aparece
200 OKSucessoGET que encontrou o recurso
201 CreatedCriadoPOST que criou algo
404 Not FoundNão encontradoRecurso inexistente
500 Server ErrorErro internoFalha no servidor
HTTP/1.1 200 OK
Content-Type: application/json

{ "id": 42, "nome": "Teclado", "preco": 350 }

Por que HTTP é stateless

HTTP é stateless: o servidor não lembra de você entre uma requisição e outra. Cada pedido precisa carregar tudo que o servidor precisa para entendê-lo. É como ligar para uma central de atendimento onde cada ligação cai num atendente diferente que não tem seu histórico — você precisa se identificar toda vez.

Isso parece uma limitação, mas é o que permite escalar: como nenhum servidor guarda "onde você estava", qualquer servidor pode atender qualquer requisição. O estado (quem você é, seu carrinho) é mantido por fora, via cookies, tokens ou banco — assunto de autenticação e autorização.

A web inteira é uma conversa de pedido e resposta sobre um protocolo sem memória. Entender request, response e o porquê do stateless é a fundação de todo o resto do back-end.

Conclusão

Do DNS ao render, cada acesso é uma troca de mensagens entre cliente e servidor sobre HTTP. Saber dissecar uma requisição (método, caminho, headers, body) e uma resposta (status, headers, body), e entender por que HTTP é stateless, é o conhecimento que sustenta APIs REST, sessões e escalabilidade. Domine esse fluxo e você terá o mapa de tudo que vem depois.
