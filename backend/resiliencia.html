Resiliência: idempotência, retries e circuit breaker

Diogo Gomes  ·  6 de Junho, 2026  ·  3 min de leitura

Em sistemas distribuídos, falhas não são exceção — são rotina. A rede cai, um serviço fica lento, uma chamada estoura o timeout. Resiliência é projetar para que essas falhas inevitáveis não derrubem o sistema inteiro. Para um sênior, dominar idempotência, retries e circuit breaker é essencial — e tema frequente em entrevistas.

Falhas acontecem

A premissa que muda tudo: em escala, algo está sempre falhando. Um disco, uma instância, um link de rede. O objetivo não é evitar 100% das falhas (impossível), mas conter o impacto e se recuperar automaticamente. Resiliência é a diferença entre "um serviço lento" e "o sistema inteiro fora do ar".

Idempotência: a base do retry seguro

Uma operação é idempotente quando executá-la várias vezes produz o mesmo efeito de executá-la uma vez. Isso é crucial porque, quando uma requisição falha, você quer poder tentar de novo — mas só pode fazer isso com segurança se o retry não causar dano. Imagine cobrar um cliente: se a resposta se perdeu mas a cobrança aconteceu, um retry cego cobraria duas vezes.

A solução é a idempotency key: o cliente envia um identificador único da operação, e o servidor garante que processa cada chave apenas uma vez:

POST /cobrancas
Idempotency-Key: 7f3a-9c21

// servidor:
if jaProcessada("7f3a-9c21"):
    return resultadoSalvo("7f3a-9c21")   // nao cobra de novo
resultado = processarCobranca(...)
salvar("7f3a-9c21", resultado)
return resultado

Retry com backoff e jitter

Quando algo falha de forma transitória, retentar faz sentido. Mas retentar imediatamente e em loop pode piorar a situação — você martela um serviço já sobrecarregado. Por isso usa-se backoff exponencial: cada tentativa espera mais que a anterior (1s, 2s, 4s, 8s). E adiciona-se jitter (uma variação aleatória) para evitar que mil clientes retentem no mesmo instante exato, criando uma onda sincronizada.

espera = min(base * 2^tentativa, max) + random(0, jitter)
// tentativa 0: ~1s    tentativa 1: ~2s
// tentativa 2: ~4s    tentativa 3: ~8s

E sempre com timeout: nunca espere indefinidamente por uma resposta. Sem timeout, uma dependência lenta trava suas threads e o problema se propaga. Lembre também que retry só é seguro em operações idempotentes.

Circuit breaker

Se um serviço está claramente fora do ar, continuar tentando (mesmo com backoff) só desperdiça recursos e atrasa respostas. O circuit breaker funciona como um disjuntor elétrico: depois de muitas falhas seguidas, ele "abre" e passa a falhar rápido, sem nem tentar a chamada — dando tempo para o serviço se recuperar.

   sucesso                 muitas falhas
  +---------+   falhas    +---------+
  | FECHADO |  -------->  |  ABERTO |  (falha rapido,
  +---------+             +---------+   nem chama)
       ^                       |
       | sucesso               | apos timeout
       |                       v
       |  falha          +-------------+
       +---------------- | MEIO-ABERTO |  (testa 1 chamada)
                         +-------------+

Os três estados: fechado (normal, deixa passar), aberto (falha rápido após o limite de erros), e meio-aberto (após um tempo, deixa passar uma chamada de teste — se der certo, fecha; se falhar, reabre).

EstadoComportamentoTransição
FechadoChamadas passam normalmenteMuitas falhas → Aberto
AbertoFalha imediata, não chamaApós timeout → Meio-aberto
Meio-abertoTesta uma chamadaSucesso → Fechado / Falha → Aberto
Retry diz "tente de novo, com calma". Circuit breaker diz "pare de tentar, está claramente quebrado". Idempotência é o que torna o primeiro seguro. Os três trabalham juntos.

Isolamento com bulkhead

Por fim, o padrão bulkhead (numa frase): assim como um navio é dividido em compartimentos estanques para que um furo não afunde tudo, você isola recursos — pools de conexão ou threads separados por dependência — para que a falha de uma parte não consuma os recursos das outras.

Conclusão

Resiliência é projetar assumindo que tudo vai falhar. Torne operações idempotentes para permitir retries seguros, use backoff exponencial com jitter e timeouts para não agravar a carga, e proteja-se com circuit breakers e bulkheads para conter o estrago. Esse conjunto é o que mantém sistemas distribuídos de pé no mundo real.
