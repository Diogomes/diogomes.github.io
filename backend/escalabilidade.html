Escalabilidade: vertical, horizontal e load balancing

Diogo Gomes  ·  10 de Junho, 2026  ·  3 min de leitura

Escalar é fazer um sistema aguentar mais carga sem cair. Como dev sênior, você precisa não só conhecer as estratégias, mas explicar seus trade-offs e identificar o gargalo real — que quase nunca é a aplicação. Esse é um dos temas centrais de entrevistas de system design.

Vertical vs horizontal

Há duas formas de escalar. Vertical (scale up) é colocar uma máquina maior: mais CPU, mais RAM. Horizontal (scale out) é colocar mais máquinas, dividindo a carga entre elas. A analogia: escalar vertical é trocar seu carro por um caminhão; escalar horizontal é comprar vários carros. Cada um resolve um tipo de problema.

AspectoVertical (scale up)Horizontal (scale out)
ComoMáquina mais potenteMais máquinas
LimiteTeto físico do hardwarePraticamente ilimitado
ComplexidadeBaixa (não muda o código)Maior (precisa coordenar)
ResiliênciaPonto único de falhaUma máquina cai, outras seguem

Por que stateless é a chave

Escalar horizontalmente só funciona bem se os serviços forem stateless: nenhuma requisição depende de qual máquina atendeu a anterior. Se o servidor guarda estado em memória (uma sessão, por exemplo), o usuário "preso" naquela máquina não pode ser atendido por outra. Por isso o estado vai para fora — um banco, um cache compartilhado como Redis. Com serviços stateless, adicionar máquinas é trivial.

Load balancer

Com várias instâncias, alguém precisa distribuir as requisições entre elas: o load balancer. Ele fica na frente, recebe todo o tráfego e o reparte:

             +----------------+
   clientes  |  Load Balancer |
  ---------> |  (round-robin) |
             +----------------+
               /     |      \
              v      v       v
          [App 1] [App 2] [App 3]   <- instancias stateless
              \      |      /
               v     v     v
            +-------------------+
            |   Banco de dados  |   <- estado compartilhado
            +-------------------+

A estratégia mais simples é round-robin: distribui as requisições em rodízio, uma para cada instância em sequência. Outras opções consideram qual instância está menos ocupada (least connections) ou a localização do usuário.

O banco como gargalo

Aqui está o ponto que separa juniores de seniores: você pode ter 50 instâncias da aplicação, mas se todas batem no mesmo banco, o banco vira o gargalo. A aplicação escala fácil; o estado, não. As saídas principais:

  • Read replicas: cópias do banco que atendem só leituras, aliviando o banco principal (que fica com as escritas). Funciona bem quando há muito mais leitura que escrita.
  • Sharding: particionar os dados entre vários bancos (ex.: usuários A-M num, N-Z noutro), distribuindo também as escritas — porém com grande aumento de complexidade.
Escalar a aplicação é a parte fácil; escalar o estado é a parte difícil. Quando alguém diz "é só subir mais servidores", pergunte: e o banco aguenta?

Trade-offs

Não existe almoço grátis. Horizontal traz resiliência e escala quase ilimitada, mas exige serviços stateless, load balancers e lida com a complexidade de sistemas distribuídos — incluindo decisões de consistência que aparecem no teorema CAP. Vertical é simples mas tem teto e é ponto único de falha. Na prática, sistemas maduros combinam os dois: escalam vertical até certo ponto e horizontal a partir daí.

Conclusão

Escalabilidade é sobre remover gargalos, não sobre adicionar máquinas cegamente. Mantenha serviços stateless, use um load balancer para distribuir carga e, sobretudo, lembre que o banco costuma ser o limite real. Saber onde está o gargalo e qual trade-off aceitar é exatamente o tipo de raciocínio esperado de um sênior.
