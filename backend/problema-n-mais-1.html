O problema N+1: como o ORM te trai

Diogo Gomes  ·  15 de Junho, 2026  ·  3 min de leitura

O problema N+1 é uma das armadilhas de performance mais comuns no back-end, e o mais traiçoeiro é que o código parece perfeito. Ele nasce da comodidade dos ORMs e só aparece quando o volume de dados cresce. Saber identificá-lo e resolvê-lo é conhecimento esperado de qualquer dev pleno.

O que é o problema N+1

O nome descreve exatamente o que acontece: você faz 1 query para buscar uma lista de itens, e depois N queries — uma para cada item — para buscar dados relacionados. Se você lista 100 pedidos e, para cada um, busca o cliente, são 1 + 100 = 101 queries onde duas bastariam.

Como o ORM te trai

ORMs (Object-Relational Mappers) deixam acessar relações como se fossem atributos comuns. O problema é que cada acesso pode disparar uma query escondida — o chamado lazy loading:

pedidos = Pedido.all()          // 1 query: SELECT * FROM pedidos

for pedido in pedidos:
    print(pedido.cliente.nome)  // N queries! cada .cliente
                                // dispara SELECT * FROM clientes WHERE id = ?

O código é limpo e legível — essa é a traição. Nada indica visualmente que pedido.cliente está indo ao banco a cada iteração. Com 10 pedidos você nem percebe; com 10 mil, a página trava.

Como detectar

A forma mais direta é ligar o log de queries do ORM em ambiente de desenvolvimento. Se você vê dezenas de SELECTs quase idênticos variando só o ID, encontrou um N+1:

SELECT * FROM pedidos;
SELECT * FROM clientes WHERE id = 1;
SELECT * FROM clientes WHERE id = 2;
SELECT * FROM clientes WHERE id = 3;
... (repete N vezes)

No dia a dia isso aparece quando um endpoint que era rápido começa a degradar conforme a base cresce. Ferramentas de APM e o próprio log de queries são seus aliados.

Como resolver: eager loading

A solução é dizer ao ORM, de antemão, que você vai precisar da relação. Isso é o eager loading, que o ORM traduz em um JOIN ou em uma segunda query em lote (batch):

// Antes (N+1):
pedidos = Pedido.all()

// Depois (eager loading):
pedidos = Pedido.all().include('cliente')

// O ORM gera algo como:
// SELECT * FROM pedidos;
// SELECT * FROM clientes WHERE id IN (1, 2, 3, ...);  // 1 query so

Antes e depois: o impacto

CenárioQueries (100 itens)Resultado
Lazy loading (N+1)101Lento, escala mal
Eager loading (JOIN ou IN)1 ou 2Rápido e constante

A diferença não é só velocidade: o N+1 piora linearmente com o número de itens, enquanto o eager loading mantém o custo praticamente constante. É a diferença entre um endpoint que aguenta crescimento e outro que vira gargalo.

O N+1 não é um bug do ORM — é o ORM fazendo exatamente o que você pediu sem perceber. A solução é tornar a intenção explícita: avise que vai precisar da relação.

Quando aceitar e cuidados

Nem todo loop com acesso a relação é um problema: se a lista tem 2 itens, o N+1 é irrelevante. Por outro lado, eager loading exagerado (carregar relações que você não usa) também desperdiça recursos. O equilíbrio é carregar o que você de fato vai usar, em lote. Esse tema conversa diretamente com cache no back-end, outra alavanca de performance.

Conclusão

O problema N+1 é o exemplo perfeito de como abstrações cômodas escondem custos. Aprenda a ler o log de queries, reconheça o padrão de SELECTs repetidos e use eager loading para colapsar N queries em uma. É um daqueles conhecimentos que, uma vez aprendidos, você passa a enxergar em todo lugar.
