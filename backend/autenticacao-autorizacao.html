Autenticação vs Autorização: sessions, JWT e OAuth

Diogo Gomes  ·  17 de Junho, 2026  ·  3 min de leitura

Autenticação e autorização são confundidas o tempo todo, mas resolvem problemas diferentes. Como dev pleno, você precisa não só saber a diferença, mas escolher entre sessões e tokens, entender a estrutura de um JWT e explicar onde OAuth2 entra. Esses temas aparecem em quase toda entrevista de back-end.

Authn vs Authz

Autenticação (authn) responde "quem é você?". Autorização (authz) responde "o que você pode fazer?". A analogia: autenticação é mostrar seu RG na portaria de um prédio; autorização é o crachá que diz a quais andares você pode subir. Você precisa provar quem é antes de descobrir o que pode acessar — primeiro authn, depois authz.

Sessão com cookie

No modelo de sessão, o usuário faz login, o servidor cria uma sessão e guarda o estado dela (em memória, banco ou Redis). Ele devolve um cookie com um ID de sessão. A cada requisição, o navegador envia o cookie, e o servidor procura a sessão correspondente:

login OK -> servidor guarda sessao abc123 { userId: 7 }
        -> Set-Cookie: session=abc123

requisicao -> Cookie: session=abc123
           -> servidor busca abc123, sabe que e o user 7

O ponto-chave: o estado vive no servidor. Isso facilita revogar acessos, mas exige que todos os servidores compartilhem onde a sessão está guardada.

Token JWT

Com JWT (JSON Web Token), o servidor não guarda estado. No login ele gera um token assinado que contém os dados do usuário, e o cliente o envia em cada requisição (geralmente no header Authorization: Bearer ...). O servidor só verifica a assinatura — não precisa consultar nada. Por isso JWT é stateless, o que combina com APIs REST e escala bem horizontalmente.

A estrutura de um JWT

Um JWT tem três partes separadas por ponto: header.payload.signature.

header    -> { "alg": "HS256", "typ": "JWT" }
payload   -> { "sub": 7, "role": "admin", "exp": 1735689600 }
signature -> HMAC(base64(header) + "." + base64(payload), SEGREDO)

Header e payload são apenas base64, não criptografia — qualquer um consegue ler. A assinatura garante que o conteúdo não foi adulterado, porque só quem tem o segredo consegue gerá-la. Daí a regra de ouro: nunca coloque segredos no payload (senhas, dados sensíveis), pois ele é legível.

Sessão vs JWT

AspectoSessão (cookie)JWT (token)
EstadoNo servidor (stateful)No token (stateless)
Escala horizontalExige store compartilhadoNatural, sem store
RevogaçãoFácil (apaga a sessão)Difícil (token vale até expirar)
Tamanho enviadoPequeno (só o ID)Maior (carrega os dados)

OAuth2: autorização delegada

OAuth2 resolve outro problema: deixar um app acessar seus dados em outro serviço sem entregar sua senha. É o que acontece no "login com Google". Você é redirecionado ao Google, autentica lá, e o Google devolve ao app um token que autoriza apenas o que você consentiu. O app nunca vê sua senha. Importante: OAuth2 é sobre autorização delegada, não sobre login — embora seja a base de fluxos de login social.

Autenticação prova quem você é; autorização decide o que você faz. Sessão guarda o estado no servidor; JWT carrega o estado no token. Confundir esses pares é o erro mais comum da área.

Notas de segurança

Pontos que sempre valem citar: defina expiração curta nos tokens (claim exp) e use refresh tokens para renovar; use HTTPS sempre; marque cookies como HttpOnly e Secure; e lembre que JWT revogado antes da hora é um problema real — por isso tokens de vida longa são arriscados.

Conclusão

Authn e authz são camadas distintas e complementares. Sessões oferecem controle fácil de revogação; JWTs oferecem statelessness e escala. OAuth2 cuida do acesso delegado. Saber quando usar cada um, e os cuidados de segurança de cada escolha, é exatamente o que se espera de um pleno.
